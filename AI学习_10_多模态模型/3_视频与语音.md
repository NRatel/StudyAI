## 1. 视频和语音多了哪些麻烦
### 1.1 从图像到视频：多了什么？

> **重要区分**：在进入视频多模态之前，请先明确两个完全不同的问题方向：
> - **视频理解**（输入视频 → 输出文字）：给模型一段视频，让它描述内容、回答问题、提取信息。核心挑战是时空建模——如何同时理解每帧的空间内容和帧间的时间动态。代表：VideoMAE、InternVideo、Video-LLaVA。
> - **视频生成**（输入文字 → 输出视频）：给模型一段文字描述，让它生成对应的视频。核心挑战是时间一致性——如何确保生成的每帧之间连贯、物体不突变。代表：Sora、CogVideoX、Wan。
>
> 这两个方向的技术栈几乎完全不同（理解侧基于视觉 Transformer，生成侧基于扩散模型），不要混为一谈。本节先讲视频理解（2.1），再讲视频生成（2.2），最后讲语音（2.3-2.4）。

前两节我们学了图像的多模态建模（ViT、CLIP、LLaVA）。视频在图像的基础上多了一个维度——**时间**。

| 模态 | 数据形状 | 本质 |
|------|---------|------|
| 图像 | (H, W, C) | 2D 空间信号 |
| 视频 | (T, H, W, C) | 2D 空间 + 1D 时间 = 3D 时空信号 |
| 语音 | (T, F) | 1D 时间 + 频率 = 2D 时频信号 |

视频理解的核心挑战是**时空建模**——不仅要理解每一帧"有什么"（空间），还要理解帧与帧之间"发生了什么"（时间）。

类比：图像理解是看一张照片，视频理解是看一段电影。你不仅要识别场景中的物体，还要理解动作、事件、因果关系。

### 1.2 语音的特殊性

语音与文本/图像有本质区别：

```
文本：  离散 token → 天然适合 Transformer
图像：  2D 像素 → ViT 切 patch → token 化
语音：  连续波形 → 需要先转为频谱图（Spectrogram）→ 再处理

语音波形 (1D, 16kHz)
    │
    ▼
短时傅里叶变换 (STFT)
    │
    ▼
梅尔频谱图 (2D: 时间 × 频率)  ← 类似于"图像"，可以用类似方法处理
    │
    ▼
Transformer 编码
```

## 2. 视频、语音模型的核心机制
### 2.1 视频理解：时空建模

#### 2.1.1 从图像 Transformer 到视频 Transformer

图像 ViT 处理的是 2D patch 序列。视频有额外的时间维度，需要处理 3D "时空管"（spatiotemporal tube）。

**视频 token 化**：

```
视频输入: (T, H, W, C)
    │
    ▼
切分为时空 patch (tubelet):
    每个 patch 覆盖 t × h × w 的时空区域
    │
    ▼
Token 数量: N = (T/t) × (H/h) × (W/h)
    │
    例如: 16 帧 × 224×224 图像, tubelet=2×16×16
    N = (16/2) × (224/16) × (224/16) = 8 × 14 × 14 = 1568 个 token
```

**核心问题**：1568 个 token 的全局注意力计算量是 $O(1568^2) \approx 250$ 万——对于更长的视频，这个数字会爆炸。

#### 2.1.2 时空注意力的三种策略

```
策略 1: 联合时空注意力（Joint Space-Time）
┌─────────────────────────┐
│ 所有时空 token 做全局注意力 │
│ 计算量: O((T×H×W)^2)     │  ← 最强但最贵
│ 代表: ViViT Model 1      │
└─────────────────────────┘

策略 2: 分解时空注意力（Divided Space-Time）
┌─────────────────────────┐
│ 先做空间注意力（每帧内部） │
│ 再做时间注意力（跨帧同位置）│
│ 计算量: O(T×(HW)^2 + HW×T^2) │  ← 大幅降低
│ 代表: TimeSformer          │
└─────────────────────────┘

策略 3: 窗口化注意力
┌─────────────────────────┐
│ 在局部时空窗口内做注意力   │
│ 类似 Swin Transformer    │
│ 计算量: O(T×H×W×w^3)     │  ← w 是窗口大小
│ 代表: Video Swin          │
└─────────────────────────┘
```

**TimeSformer 的分解注意力**：

```
输入: 视频 token 序列 (T 帧, 每帧 N 个空间 token)

每个 Transformer Block 内部:
┌──────────────────────────────────┐
│ Step 1: 时间注意力                 │
│   对每个空间位置，沿时间维度做注意力 │
│   (同一位置在不同帧之间交互)        │
│                                   │
│ Step 2: 空间注意力                 │
│   对每帧，在空间维度做注意力         │
│   (同一帧内的不同位置交互)          │
└──────────────────────────────────┘
```

形式化地，对位于时间 $t$、空间位置 $(i,j)$ 的 token $\mathbf{z}_{t,i,j}$：

$$\mathbf{z}_{t,i,j}' = \text{TemporalAttn}(\mathbf{z}_{t,i,j};\; \{\mathbf{z}_{t',i,j}\}_{t'=1}^{T}) \quad \text{(时间注意力：与同位置所有帧交互)}$$

$$\mathbf{z}_{t,i,j}'' = \text{SpatialAttn}(\mathbf{z}_{t,i,j}';\; \{\mathbf{z}_{t,i',j'}'\}_{i',j'}) \quad \text{(空间注意力：与同帧所有位置交互)}$$

#### 2.1.3 VideoMAE：视频自监督预训练

VideoMAE（Video Masked Autoencoder, 2022）将 MAE 的掩码预训练思路扩展到视频：

```
原始视频 → 切分为时空 patch → 随机掩码 90%~95% 的 patch
                                       │
                ┌──────────────────────┘
                │
        可见 patch (5%~10%)
                │
                ▼
          ViT Encoder
                │
                ▼
     Encoder 输出 + 掩码 token
                │
                ▼
          ViT Decoder
                │
                ▼
        重建被掩码的 patch
```

**关键发现**：视频中相邻帧高度相似（时间冗余），因此需要**极高的掩码率**（90-95%，而图像 MAE 只需 75%），迫使模型学习时间动态而非简单复制邻近帧。

### 2.2 Sora：视频生成的里程碑

#### 2.2.1 Sora 的技术架构

Sora（OpenAI, 2024.2 发布技术报告）将扩散模型和 Transformer 结合，实现高质量长视频生成。

```
文本 prompt: "一只猫在钢琴上弹奏"
        │
        ▼
  文本编码器 (推测: T5 或 CLIP)
        │
        ▼ 文本 embedding
        │
        ├──────────────────────────────┐
        │                              │
        ▼                              │
┌─────────────────────┐                │
│ 视频 VAE Encoder     │                │
│                     │                │
│ 视频 → 潜空间表示    │                │
│ (T,H,W,C) →        │                │
│ (t,h,w,c)           │  ← 大幅压缩    │
│                     │    时空分辨率    │
└────────┬────────────┘                │
         │                             │
         ▼                             │
┌─────────────────────┐                │
│ 时空 Patch 化        │                │
│                     │                │
│ 潜空间 → token 序列  │                │
│ (t,h,w,c) →        │                │
│ (N, d) 个 token     │                │
└────────┬────────────┘                │
         │                             │
         ▼                             │
┌──────────────────────────────────┐   │
│        DiT (Diffusion Transformer)│   │
│                                   │   │
│  输入: 噪声 token + 时间步 t       │   │
│  条件: 文本 embedding  ←──────────┼───┘
│                                   │
│  重复 T 步去噪:                    │
│  z_t → DiT → z_{t-1} → ... → z_0 │
│                                   │
└────────┬─────────────────────────┘
         │
         ▼
┌─────────────────────┐
│ 视频 VAE Decoder     │
│                     │
│ 潜空间 → 高清视频    │
│ (t,h,w,c) →        │
│ (T,H,W,C)           │
└─────────────────────┘
```

#### 2.2.2 Sora 的关键技术

| 技术 | 说明 |
|------|------|
| **时空 VAE** | 将高分辨率视频压缩到低维潜空间，同时在时间和空间维度压缩 |
| **时空 Patch** | 在潜空间中切 3D patch，而非在像素空间。极大减少 token 数 |
| **DiT 架构** | 用 Transformer 替代 U-Net 作为扩散模型的去噪骨干网络（参考模块 06） |
| **可变分辨率/时长** | 通过不固定 token 数量，支持任意分辨率和时长的视频生成 |
| **Scaling** | 推测使用了海量视频数据 + 大规模模型 + 长时间训练 |

#### 2.2.3 DiT 在视频生成中的应用

回顾 DiT（Diffusion Transformer，Peebles & Xie, 2023）的核心思想：

**图像 DiT**：

$$\mathbf{z}_{t-1} = \text{DiT}(\mathbf{z}_t, t, \mathbf{c})$$

其中 $\mathbf{z}_t$ 是第 $t$ 步的噪声潜表示，$t$ 是时间步，$\mathbf{c}$ 是文本条件。

**视频 DiT（Sora 的扩展）**：

$$\mathbf{Z}_{t-1} = \text{VideoDiT}(\mathbf{Z}_t, t, \mathbf{c})$$

其中 $\mathbf{Z}_t \in \mathbb{R}^{N_{st} \times d}$ 是时空 patch 的噪声序列，$N_{st} = N_t \times N_h \times N_w$ 是时空 token 总数。

VideoDiT 中的注意力可以是全局的（所有时空 token 互相关注）或分解的（先空间后时间），取决于视频长度和计算预算。

**条件注入**使用 AdaLN-Zero（Adaptive Layer Normalization）：

$$\gamma, \beta = \text{MLP}([\text{emb}(t),\; \mathbf{c}])$$
$$\hat{\mathbf{h}} = \gamma \odot \text{LayerNorm}(\mathbf{h}) + \beta$$

### 2.3 Whisper：通用语音识别

#### 2.3.1 Whisper 的架构

Whisper（Radford et al., 2022）是 OpenAI 发布的通用语音识别模型，采用标准的 Encoder-Decoder Transformer 架构：

```
语音输入 (16kHz 波形，最长 30 秒)
    │
    ▼
┌──────────────────────────┐
│ 预处理: 梅尔频谱图        │
│                          │
│ 波形 → STFT → 80 维梅尔  │
│ 频谱图 (80, T_mel)       │
│                          │
│ T_mel = 3000 (30秒音频)  │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│ 2 层 1D 卷积              │
│ (stride=2, 下采样 2 倍)    │
│                          │
│ (80, 3000) → (D, 1500)   │
│ + 正弦位置编码             │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│ Transformer Encoder       │
│ (N_enc 层)                │
│                          │
│ 输出: 音频特征序列         │
│ (1500, D)                │
└────────┬─────────────────┘
         │
         ▼
┌──────────────────────────┐
│ Transformer Decoder       │
│ (N_dec 层)                │
│                          │
│ 自回归生成文本 token       │
│ 支持多种任务:              │
│  - 语音转录                │
│  - 语音翻译                │
│  - 语言检测                │
│  - 时间戳标注              │
└──────────────────────────┘
```

#### 2.3.2 Whisper 的多任务设计

Whisper 的巧妙之处在于用**特殊 token 序列**指定不同任务：

```
Decoder 输入格式:

<|startoftranscript|>    ← 开始标记
<|en|>                   ← 语言标记（自动检测或指定）
<|transcribe|>           ← 任务标记（转录 or 翻译）
<|notimestamps|>         ← 是否输出时间戳
Hello, how are you?      ← 生成的文本
<|endoftext|>            ← 结束标记

任务切换示例:
  语音转录: <|zh|><|transcribe|> → 输出中文文本
  语音翻译: <|zh|><|translate|>  → 输出英文文本（将中文语音翻译为英文）
```

#### 2.3.3 Whisper 的训练数据

| 维度 | 数据 |
|------|------|
| 数据量 | 68 万小时 |
| 数据来源 | 互联网音频（自动字幕、播客等） |
| 语言数 | 99 种语言 |
| 标注方式 | 弱监督（使用已有的自动字幕作为标签） |
| 数据质量 | 远不如人工标注，但规模弥补质量 |

**关键洞察**：Whisper 证明了一个重要规律——**海量弱标注数据 > 少量精标注数据**。这与 CLIP 的思路一致（4 亿网络图文对 vs 人工标注的 ImageNet）。

### 2.4 语音 + LLM：端到端语音大模型

#### 2.4.1 传统级联方案 vs 端到端方案

```
传统级联方案（GPT-4o 之前）:
  语音 → ASR (Whisper) → 文本 → LLM → 文本 → TTS → 语音
                                        │
              延迟: 2~5 秒                │
              问题: 丢失语音中的情感/语气   │

端到端方案（GPT-4o 等）:
  语音 → 语音编码器 → 语音 token → LLM → 语音 token → 语音解码器 → 语音
                                        │
              延迟: ~320ms               │
              优势: 保留情感/语气，实时对话 │
```

#### 2.4.2 语音 token 化方案

将连续语音波形转化为离散 token 是端到端方案的关键：

| 方案 | 做法 | 代表 |
|------|------|------|
| **语义 token** | 用自监督模型（如 HuBERT）提取高层语义特征 → 量化 | GSLM、AudioPaLM |
| **声学 token** | 用神经编解码器（如 EnCodec）压缩为离散码本 | VALL-E、SoundStorm |
| **混合 token** | 语义 token + 声学 token 分层处理 | GPT-4o（推测）、CosyVoice |
| **连续特征** | 不做离散化，直接用连续特征接入 LLM | Qwen2-Audio |
