> **前置知识**：[分布式训练](1_分布式训练.md)

## 风趣易懂的直觉讲解
### 为什么要用半精度？

一个直觉：FP32 用 4 字节存一个数，FP16 只用 2 字节——**显存直接减半，计算速度翻倍**。NVIDIA 的 Tensor Core 专门为半精度矩阵乘法优化，A100 的 FP16 算力（312 TFLOPS）是 FP32（19.5 TFLOPS）的 16 倍。

但直接把所有数据类型从 FP32 换成 FP16 会出问题：

```
FP32: 1位符号 + 8位指数 + 23位尾数  → 范围 ±3.4×10³⁸,  精度 ~7位有效数字
FP16: 1位符号 + 5位指数 + 10位尾数  → 范围 ±6.5×10⁴,   精度 ~3位有效数字
BF16: 1位符号 + 8位指数 + 7位尾数   → 范围 ±3.4×10³⁸,  精度 ~2位有效数字
```

FP16 的问题：
- **下溢**：梯度值小于 `5.96e-8`（FP16 最小正数）就变成 0
- **上溢**：值超过 65504 就变成 inf
- **精度损失**：参数更新量 `lr × grad` 如果太小，加到参数上可能被"吞掉"

所以不能"纯用 FP16"，要**混合精度**。

### 混合精度的核心想法

```
计算用 FP16 (快)        ←→        参数更新用 FP32 (准)

前向/反向:  FP16 参数 → FP16 激活 → FP16 梯度     (GPU Tensor Core 加速)
                                        ↓
参数更新:   FP32 主参数 ← FP32(梯度) × lr           (CPU 或 GPU 上的精确计算)
                  ↓
下一轮:     FP32 主参数 → 转回 FP16 → 下一次前向
```

**三个关键技巧**：
1. **FP32 主权重 (Master Weights)**：保留一份 FP32 参数用于更新
2. **Loss Scaling**：放大 loss 让梯度不下溢
3. **算术精度控制**：某些操作（如 Softmax、LayerNorm）用 FP32 计算

## 准确概念定义与核心原理
### 1. 数值格式对比

| 格式 | 总位数 | 指数 | 尾数 | 范围 | 最小正数 | 适用场景 |
|------|-------|------|------|------|---------|---------|
| FP32 | 32 | 8 | 23 | ±3.4e38 | 1.2e-38 | 参数更新 |
| FP16 | 16 | 5 | 10 | ±6.5e4 | 6.0e-8 | 前向/反向计算 |
| BF16 | 16 | 8 | 7 | ±3.4e38 | 1.2e-38 | 前向/反向（更稳定） |
| TF32 | 19 | 8 | 10 | ±3.4e38 | 1.2e-38 | A100+ 默认 FP32 模式 |
| FP8 E4M3 | 8 | 4 | 3 | ±448 | 1.95e-3 | H100+ 前向 |
| FP8 E5M2 | 8 | 5 | 2 | ±57344 | 6.1e-8 | H100+ 反向 |

**BF16 vs FP16**：
- BF16 保留了 FP32 的指数范围（8位指数，与 FP32 相同），所以**不需要 Loss Scaling**，不易上溢/下溢
- FP16 精度更高（10位尾数 vs BF16 的 7位），但指数范围太小（5位指数，最大 65504），容易溢出
- **核心权衡**：BF16 范围更大不易溢出但精度略低；FP16 精度更高但范围小需要 Loss Scaling
- 现代 GPU（A100 及之后的 Ampere+ 架构）推荐使用 BF16，这也是 GPT-3 之后的主流选择

### 2. Loss Scaling 原理

问题：反向传播中，深层的梯度值可能非常小（如 1e-8），在 FP16 中会下溢为 0。

解决方案：前向结束后，把 loss 乘以一个大数（如 1024），这样反向传播的梯度都被等比放大，不会下溢。更新参数前再除以这个数。


**动态 Loss Scaling**：
- 初始 scale 设一个很大的值（如 2^16）
- 如果梯度没有 inf/nan，下次 scale × 2（尝试更大）
- 如果梯度出现 inf/nan，跳过本步更新，scale ÷ 2

### 3. DeepSpeed 框架

DeepSpeed 是微软开发的分布式训练框架，核心功能：

```
DeepSpeed
├── ZeRO 优化器 (Stage 1/2/3)
├── 混合精度训练
├── 梯度累积
├── 激活检查点 (Activation Checkpointing)
├── ZeRO-Offload (CPU/NVMe 卸载)
├── 稀疏注意力
├── 流水线并行
└── 通信优化
```

**DeepSpeed 的核心优势**：几乎不改代码，通过 JSON 配置文件控制一切。

**ZeRO-Offload / ZeRO-Infinity**：当 GPU 显存仍然不够时，把优化器状态甚至参数卸载到 CPU 内存或 NVMe SSD。

```
ZeRO-Offload (Stage 2 + CPU):
  GPU: FP16 参数 + FP16 梯度计算
  CPU: FP32 优化器状态 + 参数更新

ZeRO-Infinity (Stage 3 + CPU + NVMe):
  GPU: 当前层的 FP16 参数 (临时)
  CPU: 部分优化器状态
  NVMe: 全部优化器状态 + 参数分片
```

### 4. FSDP (Fully Sharded Data Parallel)

PyTorch 原生的 ZeRO-3 实现。核心思想与 ZeRO-3 相同：参数、梯度、优化器状态全部分片。

**FSDP vs DeepSpeed ZeRO-3**：

| 特性 | DeepSpeed ZeRO-3 | PyTorch FSDP |
|------|------------------|--------------|
| 维护方 | 微软 | Meta/PyTorch |
| 代码侵入 | 低（JSON 配置） | 中（需要包装模型） |
| 与 PyTorch 集成 | 插件 | 原生 |
| 混合精度 | 自己实现 | 用 PyTorch AMP |
| 生态 | HuggingFace 支持好 | 原生 PyTorch 生态 |
| 性能 | 略优（通信优化多） | 接近 |

### 5. 梯度累积 (Gradient Accumulation)

**问题**：显存不够放不下大 batch，但大 batch 对训练稳定性很重要。

**方案**：多次小 batch 前向反向累积梯度，攒够了再更新一次参数。

```
有效 batch_size = micro_batch_size × gradient_accumulation_steps × num_gpus

例如: 有效 batch = 4 × 8 × 64 = 2048
  - 每卡每步处理 4 条数据
  - 累积 8 步梯度
  - 64 张卡
```

数学上完全等价于用 batch_size=2048 训练。

### 6. 激活检查点 (Activation Checkpointing / Gradient Checkpointing)

**问题**：反向传播需要前向时的激活值。对于 96 层的 GPT-3，存所有激活需要巨大显存。

**方案**：只保存部分层的激活值，其他层的激活在反向传播时**重新计算**。

```
普通方式: 所有激活都存 → 显存 O(L)，计算 1x
激活检查点: 每隔 k 层存一次 → 显存 O(L/k)，计算 ~1.33x

典型配置: 每个 Transformer block 作为一个 checkpoint 单元
  - 前向: 只保存每个 block 的输入
  - 反向: 需要某 block 内部激活时，从该 block 输入重新前向计算
  - 额外计算开销: ~33% (每个 block 的前向算了两遍)
  - 显存节省: 激活值从 O(层数×隐藏维度) 降到 O(sqrt(层数)×隐藏维度)
```

## 小结

| 技术 | 核心作用 | 额外开销 | 推荐场景 |
|------|---------|---------|---------|
| BF16 混合精度 | 显存减半，速度翻倍 | 需要 FP32 主权重 | 所有训练 |
| FP16 + Loss Scaling | 同上 | Loss Scaling 复杂度 | V100 等老卡 |
| 梯度累积 | 用小 batch 模拟大 batch | 训练时间略增 | 显存不够时 |
| 激活检查点 | 显存降到 O(sqrt(L)) | ~33% 额外计算 | 深层大模型 |
| DeepSpeed | ZeRO + 混合精度 + 一键配置 | 学习配置 | 多卡训练 |
| FSDP | PyTorch 原生 ZeRO-3 | 需要包装模型 | PyTorch 原生党 |
