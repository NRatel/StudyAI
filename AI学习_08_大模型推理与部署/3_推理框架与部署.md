## 直觉与概述

### 从"模型能跑"到"模型能用"

用 HuggingFace Transformers 加载一个模型，写两行代码就能生成文本。但这和"能在生产环境中服务用户"之间有巨大的鸿沟：

```
HuggingFace 推理（单请求）:
  - 吞吐量: ~30 tokens/s (7B, A100)
  - 并发: 1 个请求
  - 显存效率: 差（大量浪费）

vLLM 推理（高并发）:
  - 吞吐量: ~1500 tokens/s (7B, A100, batch=64)
  - 并发: 数十~数百个请求
  - 显存效率: 高（PagedAttention）

差距: 50x 的吞吐量
```

推理框架的核心工作就是把第 1、2 章学的优化技术（KV Cache、PagedAttention、Continuous Batching、量化）工程化，变成开箱即用的工具。

### 主流推理框架一览

```
                    推理框架选型
                        │
        ┌───────────────┼───────────────┐
        │               │               │
   高吞吐服务        极致性能       本地/边缘部署
        │               │               │
    ┌───┴───┐      TensorRT-LLM    ┌───┴───┐
   vLLM    TGI                  llama.cpp  Ollama
```

| 框架 | 开发者 | 核心优势 | 适用场景 |
|------|--------|---------|---------|
| vLLM | UC Berkeley | PagedAttention + Continuous Batching | 高吞吐在线服务 |
| TGI | HuggingFace | 与 HF 生态深度集成 | HuggingFace 用户 |
| TensorRT-LLM | NVIDIA | 极致编译优化 + NVIDIA 硬件深度适配 | 低延迟、NVIDIA GPU |
| llama.cpp | Georgi Gerganov | CPU/边缘推理、GGUF 量化 | 本地部署、消费级硬件 |
| Ollama | Ollama Inc. | 一行命令跑模型 | 个人开发者、快速体验 |

## 严谨定义与原理

### 1. vLLM — 高吞吐推理的事实标准

#### 核心技术栈

vLLM 之所以比朴素推理快 2~24 倍，是因为它集成了本模块前两章的几乎所有优化：

```
vLLM 技术栈:
  ├── PagedAttention (第1章)     → KV Cache 零浪费
  ├── Continuous Batching (第1章) → GPU 利用率最大化
  ├── 量化支持 (第2章)            → GPTQ, AWQ, FP8
  ├── CUDA Kernel 优化            → Flash Attention, 自定义算子
  ├── Tensor Parallelism          → 多 GPU 分布式推理
  └── Prefix Caching              → 系统 prompt 复用
```

#### vLLM 的调度器

vLLM 的调度器是其高吞吐的关键，它需要在每一步做出以下决策：

```
每一步 decode:
  1. 哪些请求已完成? → 移出 batch，释放 KV Cache 块
  2. 有新请求到达? → 尝试加入 batch
  3. 显存够吗? → 如果不够，是否需要 preempt（暂停某些请求）
  4. 有 prefill 请求? → 和 decode 请求混合调度

调度策略:
  - First-Come-First-Served (FCFS): 默认，公平
  - 优先短请求: 适合 SLA 敏感场景
  - Preemption: 显存不足时暂停低优先级请求，释放 KV Cache
```

#### vLLM 的 API

vLLM 提供与 OpenAI 兼容的 API，可以无缝替换：

```python
# 启动 vLLM 服务
# python -m vllm.entrypoints.openai.api_server \
#   --model meta-llama/Llama-2-7b-chat-hf \

#   --max-model-len 4096

# 客户端调用（与 OpenAI SDK 完全兼容）

    model="meta-llama/Llama-2-7b-chat-hf",
    messages=[{"role": "user", "content": "Hello!"}],

```

### 2. TGI (Text Generation Inference) — HuggingFace 官方推理

TGI 是 HuggingFace 官方的推理框架，用 Rust 编写（高性能），核心特点：

| 特性 | 说明 |
|------|------|
| Flash Attention | 内置支持 |
| Continuous Batching | 支持（称为 Token Streaming） |
| 量化 | GPTQ、bitsandbytes、EETQ |
| 分布式 | Tensor Parallelism |
| 水印 | 可对生成文本加水印 |
| HF Hub 集成 | 直接从 Hub 加载模型 |

**TGI vs vLLM**:
- TGI 更适合已经深度使用 HuggingFace 生态的团队
- vLLM 在纯吞吐量上通常领先（得益于 PagedAttention）
- 两者都在快速迭代，差距在缩小

### 3. TensorRT-LLM — NVIDIA 的极致优化

TensorRT-LLM 是 NVIDIA 官方的 LLM 推理解决方案，核心是**编译时优化**：

```
HuggingFace/PyTorch 推理:
  Python → PyTorch → CUDA Kernels
  ↑ 大量 Python 开销，Kernel 未优化

TensorRT-LLM:
  模型定义 → TensorRT 编译 → 优化后的 CUDA Engine
  ↑ 消除 Python 开销，算子融合，显存优化
```

**TensorRT-LLM 的核心优化**：

| 优化 | 说明 |
|------|------|
| 算子融合（Op Fusion） | 把多个小算子合并成一个大 Kernel，减少 Kernel 启动开销 |
| FP8 量化 | 在 H100/Ada 上利用硬件 FP8 单元 |
| In-flight Batching | 类似 Continuous Batching，NVIDIA 版本 |
| Paged KV Cache | 类似 PagedAttention |
| 多 GPU 并行 | Tensor Parallel + Pipeline Parallel |

**优势**：在 NVIDIA GPU 上通常能达到最低延迟。

**劣势**：编译过程复杂、对模型修改不灵活、与 NVIDIA 硬件深度绑定。

### 4. llama.cpp / GGML — CPU 推理的王者

llama.cpp（Georgi Gerganov, 2023）是一个纯 C/C++ 实现的 LLM 推理框架，专为**低资源环境**设计：

```
llama.cpp 核心特点:
  - 纯 C/C++，无 Python/PyTorch 依赖
  - 支持 CPU 推理（AVX2/ARM NEON 优化）
  - 也支持 GPU（CUDA/Metal/Vulkan）
  - GGUF 量化格式：2~8 bit 灵活选择
  - 极低内存占用：7B Q4 模型只需 ~4GB RAM
```

**GGUF 格式的生态**：

```
模型发布流程:
  原始模型 (FP16) → 转换为 GGUF → 提供多种量化版本

HuggingFace 上常见:
  TheBloke/Llama-2-7B-GGUF/
    ├── llama-2-7b.Q2_K.gguf    (2.87 GB, 最小)
    ├── llama-2-7b.Q4_K_M.gguf  (4.08 GB, 推荐)
    ├── llama-2-7b.Q5_K_M.gguf  (4.78 GB, 精度更好)
    ├── llama-2-7b.Q6_K.gguf    (5.53 GB, 接近无损)
    └── llama-2-7b.Q8_0.gguf    (7.16 GB, 几乎无损)
```

### 5. Prompt Engineering — 为什么提示词有效

Prompt Engineering 看起来像是"黑魔法"，但它的有效性有深层原因。

#### 为什么提示词能影响模型输出？

**根本原因：语言模型是条件概率生成器**

$$P(\text{output} | \text{prompt}) = \prod_{t=1}^{T} P(x_t | x_1, ..., x_{t-1}, \text{prompt})$$

改变 prompt 就是改变了条件概率的条件，模型会到概率空间的不同区域去采样：

```
Prompt A: "写一首诗"
  → 模型进入"诗歌"概率区域
  → 输出押韵、有意境的文字

Prompt B: "作为一个资深Python工程师，解释这段代码"
  → 模型进入"技术专家"概率区域
  → 输出专业、结构化的解释

Prompt C: "一步一步地思考这个数学问题"
  → 模型进入"推理"概率区域 (Chain of Thought)
  → 输出详细的推理过程，而不是直接给答案
```

#### 为什么 In-Context Learning 有效？

关于 ICL 为什么有效，有三个主流假说（详见模块 05）：

1. **隐式贝叶斯推理**：few-shot 示例帮助模型缩小任务空间
2. **格式锚定**：示例定义了输入-输出的格式，模型只需"模仿"
3. **隐式梯度下降**：注意力机制在前向传播中执行类似参数更新的操作

#### 常见 Prompt 技巧及其原理

| 技巧 | 原理 | 效果 |
|------|------|------|
| Chain of Thought (CoT) | 强制模型输出中间步骤，避免"跳步" | 数学推理准确率从 ~20% 提升到 ~80% |
| Few-shot | 在 prompt 中给出示例，锚定任务格式 | 大幅提升任务准确率 |
| Role Playing | 设定角色改变模型的先验分布 | 输出风格和专业度改变 |
| Self-Consistency | 多次采样取多数投票 | 推理准确率进一步提升 |
| ReAct | 交替 Reasoning 和 Acting | 让模型能使用外部工具 |

#### CoT 为什么对推理任务有效的深层原因

```
直接回答: "36 × 17 = ?"
  → 模型需要一步到位算出 612
  → 对于预训练时没见过的大数乘法，容易出错

CoT: "36 × 17, 让我们一步一步计算:"
  → "36 × 17 = 36 × 10 + 36 × 7"
  → "36 × 10 = 360"
  → "36 × 7 = 252"
  → "360 + 252 = 612"
  → 每一步都是简单运算，大模型都能做对
  → 相当于把难题分解成多个简单题

本质: CoT 把"一步推理"变成"多步序列化推理"
      而自回归模型天然擅长序列化生成
```

### 示例 1：vLLM 推理 vs HuggingFace 推理吞吐对比

```python
"""
对比 vLLM 和 HuggingFace Transformers 的推理吞吐量。

注意: 实际运行需要 GPU 和对应的 pip 包:
  pip install vllm transformers torch

以下代码展示了完整的对比流程。
"""
import time

def benchmark_huggingface(model_name, prompts, max_tokens=128):
    """HuggingFace Transformers 推理（基线）"""
    from transformers import AutoModelForCausalLM, AutoTokenizer
    import torch

    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModelForCausalLM.from_pretrained(
        model_name, torch_dtype=torch.float16, device_map="auto"
    )
    tokenizer.pad_token = tokenizer.eos_token

    total_tokens = 0
    t0 = time.time()

    for prompt in prompts:
        inputs = tokenizer(prompt, return_tensors="pt").to("cuda")
        with torch.no_grad():
            outputs = model.generate(
                **inputs,
                max_new_tokens=max_tokens,
                do_sample=False,
            )
        total_tokens += outputs.shape[1] - inputs["input_ids"].shape[1]

    elapsed = time.time() - t0
    return {
        "framework": "HuggingFace",
        "total_tokens": total_tokens,
        "elapsed_s": elapsed,
        "tokens_per_s": total_tokens / elapsed,
    }

def benchmark_vllm(model_name, prompts, max_tokens=128):
    """vLLM 推理（优化版）"""
    from vllm import LLM, SamplingParams

    llm = LLM(model=model_name, dtype="float16")
    params = SamplingParams(max_tokens=max_tokens, temperature=0)

    t0 = time.time()
    outputs = llm.generate(prompts, params)  # 所有请求并行处理!
    elapsed = time.time() - t0

    total_tokens = sum(len(o.outputs[0].token_ids) for o in outputs)
    return {
        "framework": "vLLM",
        "total_tokens": total_tokens,
        "elapsed_s": elapsed,
        "tokens_per_s": total_tokens / elapsed,
    }
```

```python
"""
用 llama-cpp-python 在纯 CPU 上运行 GGUF 模型。

安装: pip install llama-cpp-python
模型: 从 HuggingFace 下载 GGUF 文件
      如 TheBloke/Llama-2-7B-Chat-GGUF

    print(output["choices"][0]["text"])

    # ChatML 格式对话
    output = llm.create_chat_completion(
        messages=[
            {"role": "system", "content": "You are a helpful assistant."},
            {"role": "user", "content": "Explain KV Cache in one sentence."},
        ],
        max_tokens=100,
    )
    print(output["choices"][0]["message"]["content"])

    # 性能统计
    # 典型: 7B Q4_K_M on 8-core CPU → ~10-15 tokens/s
    # 典型: 7B Q4_K_M on M2 MacBook → ~20-30 tokens/s (Metal 加速)

# demo_llama_cpp()  # 取消注释运行（需要下载模型文件）

# ========== 不需要模型文件的演示：GGUF 文件大小计算 ==========
def estimate_gguf_size(
    params_billion: float,
    quant_type: str,
) -> dict:
    """
    估算不同量化类型的 GGUF 文件大小。

    GGUF 文件大小 ≈ 参数量 × 每参数比特数 / 8 + 元数据开销
    """
    # 各量化类型的平均每参数比特数（含 scale 等开销）
    bits_per_param = {
        "Q2_K":   2.6,
        "Q3_K_M": 3.3,
        "Q4_0":   4.5,
        "Q4_K_M": 4.8,
        "Q5_0":   5.5,
        "Q5_K_M": 5.7,
        "Q6_K":   6.6,
        "Q8_0":   8.5,
        "F16":   16.0,
    }

    bpp = bits_per_param.get(quant_type, 4.8)
    n_params = params_billion * 1e9
    size_bytes = n_params * bpp / 8
    size_gb = size_bytes / (1024 ** 3)

    return {
        "quant": quant_type,
        "bits_per_param": bpp,
        "file_size_gb": size_gb,
        "ram_needed_gb": size_gb * 1.1,  # 运行时约多 10% 开销
    }

# 各量化类型大小对比
print("LLaMA-2 7B GGUF 文件大小估算:")
print(f"{'量化类型':<10} {'每参数bit':>10} {'文件大小':>10} {'最低内存':>10}")
print("-" * 45)
for qt in ["Q2_K", "Q3_K_M", "Q4_0", "Q4_K_M", "Q5_K_M", "Q6_K", "Q8_0", "F16"]:
    r = estimate_gguf_size(7.0, qt)
    print(f"{r['quant']:<10} {r['bits_per_param']:>8.1f}b {r['file_size_gb']:>8.1f} GB"
          f" {r['ram_needed_gb']:>8.1f} GB")

print("\n推荐: Q4_K_M 是日常使用的最佳平衡（精度好、体积小）")
print("      Q5_K_M 适合追求更好精度")
print("      Q2_K   适合极低资源但可接受明显精度损失")
```

### 示例 3：Prompt Engineering 实战——Chain of Thought 效果对比

```python
"""
演示 Chain of Thought (CoT) Prompt 对推理任务的影响。

可以用 OpenAI API 或任何兼容 API（vLLM 等）运行。
这里用伪代码展示原理，然后给出可执行的本地版本。
"""

# ========== 方法 1: 概念演示（不需要 API） ==========

# 典型的推理任务和不同 Prompt 的效果
examples = [
    {
        "question": "Roger has 5 tennis balls. He buys 2 more cans of 3. How many does he have?",
        "direct_prompt": "Answer with just the number: ",
        "cot_prompt": "Let's think step by step: ",
        "expected": 11,
        "direct_common_wrong": 8,  # 模型常见错误
        "cot_trace": [
            "Roger starts with 5 tennis balls.",
            "He buys 2 cans of tennis balls.",
            "Each can has 3 tennis balls.",
            "So he buys 2 × 3 = 6 tennis balls.",
            "Total = 5 + 6 = 11 tennis balls.",
            "The answer is 11."
        ],
    },
    {
        "question": "A store had 46 apples. If they sold 19 and got 27 more, how many do they have?",
        "direct_prompt": "Answer: ",
        "cot_prompt": "Let's solve this step by step: ",
        "expected": 54,
        "direct_common_wrong": 46,
        "cot_trace": [
            "Start with 46 apples.",
            "Sold 19: 46 - 19 = 27 apples remaining.",
            "Got 27 more: 27 + 27 = 54 apples.",
            "The answer is 54."
        ],
    },
]

print("Chain of Thought (CoT) 效果演示")
print("=" * 60)

for i, ex in enumerate(examples):
    print(f"\n--- 问题 {i+1} ---")
    print(f"Q: {ex['question']}")
    print(f"正确答案: {ex['expected']}")
    print()
    print(f"直接回答 (常见错误): {ex['direct_common_wrong']}")
    print(f"CoT 推理过程:")
    for step in ex['cot_trace']:
        print(f"  → {step}")

# ========== 方法 2: 可执行的 Prompt 构建工具 ==========

def build_cot_prompt(question: str, examples: list = None) -> str:
    """
    构建 Chain of Thought 提示词。

    参数:
        question: 要回答的问题
        examples: Few-shot 示例列表 [(question, reasoning, answer), ...]
    """
    prompt = ""

    # Few-shot 示例
    if examples:
        for q, reasoning, ans in examples:
            prompt += f"Q: {q}\n"
            prompt += f"Let's think step by step:\n{reasoning}\n"
            prompt += f"The answer is {ans}.\n\n"

    # 实际问题
    prompt += f"Q: {question}\n"
    prompt += "Let's think step by step:\n"
    return prompt

def build_self_consistency_prompts(question: str, n_paths: int = 5) -> list:
    """
    构建 Self-Consistency 的多条推理路径。
    思路: 用高温度采样 n_paths 条不同的推理路径，取多数投票。
    """
    base = build_cot_prompt(question)
    return [base] * n_paths  # 相同 prompt，但用不同的 random seed 采样

# 演示
few_shots = [
    ("If there are 3 cars in the parking lot and 2 more arrive, how many are there?",
     "There are 3 cars. 2 more arrive. 3 + 2 = 5.",
     "5"),
]

question = "The cafeteria had 23 apples. They used 20 for lunch and bought 6 more. How many do they have?"
prompt = build_cot_prompt(question, few_shots)

print("\n\n构建的 CoT Prompt:")
print("-" * 40)
print(prompt)
print("-" * 40)

# Self-Consistency
paths = build_self_consistency_prompts(question, n_paths=5)
print(f"\nSelf-Consistency: 生成 {len(paths)} 条推理路径")
print("每条路径用不同 temperature/seed 采样，最后取多数投票")
print("研究表明这能进一步提升 5~15% 的准确率")
```

### 示例 4：推理框架性能评估脚手架

```python
"""
一个通用的推理框架性能评估工具。
支持测量: 吞吐量、首 token 延迟 (TTFT)、每 token 延迟 (TPOT)。
"""
import time
import statistics
from dataclasses import dataclass, field
from typing import List, Callable

@dataclass
class InferenceMetrics:
    """推理性能指标"""
    throughput_tok_per_s: float = 0.0     # 总吞吐量 (tokens/s)
    time_to_first_token_ms: float = 0.0   # 首 token 延迟 (ms)
    time_per_output_token_ms: float = 0.0 # 每 token 延迟 (ms)
    total_time_s: float = 0.0             # 总耗时
    total_tokens: int = 0                 # 总生成 token 数
    num_requests: int = 0                 # 请求数

    def __repr__(self):
        return (
            f"InferenceMetrics(\n"
            f"  throughput:     {self.throughput_tok_per_s:.1f} tokens/s\n"
            f"  TTFT:           {self.time_to_first_token_ms:.1f} ms\n"
            f"  TPOT:           {self.time_per_output_token_ms:.1f} ms\n"
            f"  total_time:     {self.total_time_s:.2f} s\n"
            f"  total_tokens:   {self.total_tokens}\n"
            f"  num_requests:   {self.num_requests}\n"
            f")"
        )

def benchmark_inference(
    generate_fn: Callable,  # 函数签名: (prompt: str) -> (tokens: list, ttft: float)
    prompts: List[str],
    warmup: int = 2,
) -> InferenceMetrics:
    """
    通用推理性能评估。

    参数:
        generate_fn: 生成函数，接收 prompt 返回 (token_list, time_to_first_token)
        prompts: 测试用的 prompt 列表
        warmup: 预热轮数（不计入统计）
    """
    # 预热
    for i in range(min(warmup, len(prompts))):
        generate_fn(prompts[i])

    # 正式测试
    ttfts = []
    tpots = []
    total_tokens = 0

    t0 = time.time()
    for prompt in prompts:
        tokens, ttft = generate_fn(prompt)
        n_tokens = len(tokens)
        if n_tokens > 0:
            ttfts.append(ttft)
            if n_tokens > 1:
                # TPOT = (总时间 - TTFT) / (n_tokens - 1)
                total_gen_time = sum(tokens)  # 假设 tokens 是每个 token 的耗时
                tpots.append(total_gen_time / max(n_tokens - 1, 1))
            total_tokens += n_tokens
    total_time = time.time() - t0

    return InferenceMetrics(
        throughput_tok_per_s=total_tokens / total_time if total_time > 0 else 0,
        time_to_first_token_ms=statistics.mean(ttfts) * 1000 if ttfts else 0,
        time_per_output_token_ms=statistics.mean(tpots) * 1000 if tpots else 0,
        total_time_s=total_time,
        total_tokens=total_tokens,
        num_requests=len(prompts),
    )

# ========== 模拟不同框架的性能 ==========
```

### 生产部署的实际考量

**考量 1: 模型选择 > 框架选择**

```
常见误区: "先选框架再选模型"
正确顺序: "先选模型（精度/成本权衡），再选最适合的框架"

例:
  - 要求最高精度 → 70B FP16 + vLLM + 4xA100
  - 要求低成本   → 7B INT4 (AWQ) + vLLM + 1xA100
  - 要求离线可用 → 7B Q4_K_M + llama.cpp + CPU
```

**考量 2: 延迟的两个维度**

```
用户感知的延迟分两部分:
  1. TTFT (Time To First Token): 用户发送请求到看到第一个字的时间
     → 主要受 Prefill 阶段影响
     → 长 prompt (如 RAG) 的 TTFT 可能很高

  2. TPOT (Time Per Output Token): 每个后续 token 的生成间隔
     → 主要受 Decode 阶段影响
     → 决定了用户看到文字"流出来"的速度

  用户体验:
    - TTFT < 500ms: 感觉流畅
    - TTFT 500ms~2s: 可以接受
    - TTFT > 2s: 明显卡顿

    - TPOT < 50ms (~20 tok/s): 比阅读速度快，流畅
    - TPOT 50~100ms (10~20 tok/s): 可以接受
    - TPOT > 100ms (< 10 tok/s): 明显缓慢
```

**考量 3: Prompt Engineering 的工程实践**

```
系统 prompt 优化:
  - 保持简洁（每多 100 token prompt → TTFT 增加 ~10ms）
  - 使用 Prefix Caching: 所有请求共享系统 prompt 的 KV Cache
  - 把不变的指令放在 prompt 开头（利于缓存）

输出控制:
  - 用 JSON mode / Structured Output 约束格式
  - 设置合理的 max_tokens（过大浪费 KV Cache 预分配）
  - 用 stop_sequences 及时停止生成
```

### 部署架构参考

```
典型的生产部署架构:

  用户请求
     │

 实例1  实例2
  │     │
  └──┬──┘
     │
     ▼
  模型权重存储 (共享文件系统 / S3)

扩展策略:
  - 水平扩展: 增加 vLLM 实例数
  - 垂直扩展: 每实例增加 GPU（Tensor Parallel）
  - 混合: 热门模型多实例，冷门模型按需启动
```

### Prompt Engineering 的常见陷阱

```
陷阱 1: Prompt 太长导致 KV Cache 爆炸
  → 10K token 的系统 prompt × 100 并发 = 大量 KV Cache 显存
  → 解决: Prefix Caching, 精简 prompt

陷阱 2: 过度依赖 Prompt 而非微调
  → 复杂任务用 prompt 反复调试不如做一次 SFT
  → 经验法则: 如果 prompt > 2000 字还效果不好，考虑微调

陷阱 3: 忽略采样参数
  → temperature=0 对事实性问题好，对创意任务差
  → top_p=0.9 通常比纯 temperature 更稳定
  → 不同任务需要不同的采样策略
```

## 本章小结

| 概念 | 一句话总结 |
|------|-----------|
| vLLM | PagedAttention + Continuous Batching，高吞吐在线服务的首选 |
| TGI | HuggingFace 官方推理框架，与 HF 生态深度集成 |
| TensorRT-LLM | NVIDIA 编译优化，极致低延迟，但部署复杂 |
| llama.cpp | 纯 C++ 实现 + GGUF 量化，CPU/边缘设备推理之王 |
| TTFT vs TPOT | 首 token 延迟和每 token 延迟是两个独立的优化维度 |
| Prompt 有效性 | 改变 prompt = 改变条件概率 = 引导模型到不同概率区域 |
| CoT | 把一步推理分解为多步序列化推理，匹配自回归生成特性 |
| 选型原则 | 高吞吐用 vLLM，低延迟用 TensorRT-LLM，本地用 llama.cpp |
