# 推理框架与部署

> **前置知识**：[KV Cache 与推理优化](1_KVCache与推理优化.md)、[量化技术](2_量化技术.md)
>
> **本节目标**：理解 vLLM、TensorRT-LLM 等推理框架主要在优化什么。

---

## 1. 部署不是把模型文件加载起来就结束

真实服务要同时面对很多请求：

- 有的 Prompt 很长。
- 有的只问一句。
- 有的生成很长。
- 有的很快结束。

如果调度不好，GPU 会有大量空闲和浪费。

## 2. 批处理：把请求凑在一起算

GPU 擅长大批量并行计算。单个请求一个个算，利用率很低。

推理框架会把多个请求动态合成批次，让 GPU 尽量忙起来。

难点是：不同请求长度不一样、生成速度不一样，批次需要不断调整。

## 3. KV Cache 管理是核心问题

长对话和并发请求会产生大量 KV Cache。

推理框架需要高效管理这些缓存：

- 谁还在生成。
- 谁已经结束。
- 哪些缓存可以回收。
- 显存碎片怎么减少。

vLLM 的 PagedAttention 就是围绕这个问题做优化。

## 4. 推测解码：先让小模型试写

推测解码会用一个更小、更快的模型先猜几个 token，再让大模型验证。

如果猜对了，就一次接受多个 token；猜错了再回退。

它像让助理先起草，大模型做审核。目标是减少大模型调用次数。

## 5. 部署时真正关心的指标

常见指标包括：

- 首 token 延迟。
- 每秒生成 token 数。
- 并发吞吐。
- 显存占用。
- 成本。
- 稳定性。

对用户来说，模型好不好不仅看回答质量，也看等多久、多少钱、能不能稳定服务。
