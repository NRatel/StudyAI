## 1. 直觉与概述

### 1.1 核心问题：Agent 为什么需要"记忆"？

想象你是一个处理复杂项目的工程师：

- 你需要**当前对话的上下文**来理解同事刚说的话（短期记忆）
- 你需要**过去项目的经验教训**来避免重复踩坑（长期记忆）
- 你需要一张**草稿纸**来临时记录中间计算结果（工作记忆）

LLM 天然只有一种"记忆"——上下文窗口。但这个窗口有固定大小限制，且随着对话进行会不断增长直到溢出。一个实用的 Agent 系统必须解决：**如何在有限的上下文窗口中，高效管理无限增长的信息？**

### 1.2 记忆的三种类型

```
                        Agent 记忆系统
                             │
            ┌────────────────┼────────────────┐
            ▼                ▼                ▼
       短期记忆          长期记忆          工作记忆
    (Short-term)      (Long-term)      (Working Memory)
            │                │                │
    当前对话的上下文    跨会话持久化       当前任务的
    消息历史记录       的知识和经验       中间状态和计划
            │                │                │
    实现：上下文窗口    实现：向量数据库    实现：Scratch Pad
    生命周期：单次会话  /摘要/知识图谱     /状态变量/计划文档
                      生命周期：永久      生命周期：单次任务
```

### 1.3 类比人类认知

| Agent 记忆类型 | 人类认知类比 | 特征 |
|---------------|-------------|------|
| **短期记忆** | 工作记忆 / 对话记忆 | 容量有限，当前正在处理的信息 |
| **长期记忆** | 语义记忆 + 情景记忆 | 容量大，需要检索才能使用 |
| **工作记忆** | 草稿纸 / 白板 | 临时存放中间结果，辅助当前推理 |

## 2. 严谨定义与原理

### 2.1 短期记忆：上下文窗口管理

短期记忆就是 LLM 的上下文窗口——传入模型的完整 token 序列。这是 Agent 最基本也最关键的记忆形式。

#### 上下文窗口的限制

| 模型 | 上下文窗口 | 约等于 |
|------|-----------|--------|
| GPT-4o（2024-2025） | 128K tokens | ~300 页文本 |
| Claude 3.5/Opus 4（2025-2026） | 200K tokens | ~500 页文本 |
| Gemini 1.5/2.0（2024-2026） | 1M-2M tokens | ~2500-5000 页文本 |

看似很大，但在 Agent 场景中消耗极快：
- 系统 Prompt + 工具定义：2K-5K tokens
- 每轮工具调用+返回：500-5000 tokens
- 10 轮交互后：轻松超过 50K tokens
- 长期运行的 Agent：几十轮后可能溢出

#### 上下文管理策略

**策略 1：滑动窗口（Sliding Window）**

```
保留：System Prompt + 最近 N 轮对话
丢弃：最早的对话轮次

优点：实现简单
缺点：丢失早期重要信息

消息历史：[sys, u1, a1, u2, a2, u3, a3, u4, a4, u5, a5]
                              ↓ 窗口大小 = 3 轮
保留：      [sys,                         u3, a3, u4, a4, u5, a5]
```

**策略 2：摘要压缩（Summarization）**

```
保留：System Prompt + 历史摘要 + 最近 K 轮对话
做法：定期将旧对话压缩为摘要

消息历史：[sys, u1, a1, u2, a2, u3, a3, u4, a4, u5, a5]
                              ↓ 压缩前 3 轮为摘要
保留：      [sys, summary_of(u1~a3),      u4, a4, u5, a5]
```

**策略 3：基于重要性的保留（Importance-based）**

```
为每条消息计算重要性分数，优先保留高重要性消息：
- 用户的原始需求 → 高重要性
- 工具调用的关键结果 → 高重要性
- 中间的思考过程 → 低重要性
- 失败的尝试 → 中等重要性（含教训）
```

### 2.2 长期记忆：跨会话的持久化知识

长期记忆让 Agent 能够"记住"跨会话的信息——用户偏好、历史交互、学到的经验。

#### 实现方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **向量存储** | 将文本嵌入为向量，语义检索 | 语义匹配强，模糊查询 | 精确查询弱，需要维护索引 |
| **摘要压缩** | 将历史压缩为结构化摘要 | 紧凑，容易注入上下文 | 压缩过程有信息损失 |
| **知识图谱** | 提取实体和关系，构建图结构 | 结构化，支持推理 | 构建和维护成本高 |
| **键值存储** | 按主题/类别存储为键值对 | 检索快，精确匹配 | 无语义泛化能力 |

#### 向量存储的工作流程

```
写入流程：
    新信息 → Embedding 模型 → 向量 → 存入向量数据库（附带原始文本和元数据）

检索流程：
    查询 → Embedding 模型 → 查询向量 → 向量数据库相似度搜索 → Top-K 相关记忆
                                                                    ↓
                                                            注入到 Agent 上下文
```

#### 记忆的生命周期管理

```
                 写入                            检索
新经验/信息 ─────────→ 长期记忆存储 ─────────→ 相关记忆
                          │
                     定期维护
                     ├── 去重：合并重复记忆
                     ├── 衰减：降低过时记忆的权重
                     ├── 整合：将碎片记忆合并为结构化知识
                     └── 遗忘：删除不再相关的记忆
```

### 2.3 工作记忆：Scratch Pad 与任务状态

工作记忆是 Agent 在执行当前任务时的"草稿纸"——存放中间结果、当前计划、待办事项等临时信息。

#### 典型的工作记忆内容

```json
{
    "current_plan": {
        "goal": "为用户生成季度数据分析报告",
        "steps": [
            {"id": 1, "desc": "获取销售数据", "status": "completed", "result": "Q1 销售额 500 万"},
            {"id": 2, "desc": "数据清洗和分析", "status": "in_progress"},
            {"id": 3, "desc": "生成可视化图表", "status": "pending"},
            {"id": 4, "desc": "撰写分析报告", "status": "pending"}
        ]
    },
    "intermediate_results": {
        "raw_data_rows": 15234,
        "cleaned_data_rows": 14891,
        "key_metrics": {"revenue": 5000000, "growth_rate": 0.12}
    },
    "notes": [
        "用户要求重点关注华东地区",
        "数据中发现 3 月有异常峰值，需要核实"
    ]
}
```

#### 工作记忆 vs 短期记忆

| 维度 | 短期记忆（上下文） | 工作记忆（Scratch Pad） |
|------|-------------------|------------------------|
| 内容 | 完整对话历史 | 提炼后的关键状态 |
| 格式 | 自然语言对话 | 结构化数据（JSON/表格） |
| 更新 | 追加式（新消息入队） | 覆盖式（状态更新） |
| 作用 | 维持对话连贯性 | 辅助复杂任务推理 |

## 3. Python 代码实践

### 3.1 上下文窗口管理器

```python
"""
上下文窗口管理器：实现滑动窗口 + 摘要压缩的混合策略
"""
import tiktoken
from openai import OpenAI
from typing import Optional

client = OpenAI()

class ContextWindowManager:
    """
    管理 Agent 的上下文窗口，防止溢出。
    策略：保留 system prompt + 历史摘要 + 最近 N 轮完整对话。
    """

    def __init__(
        self,
        model: str = "gpt-4o",
        max_tokens: int = 128_000,
        reserve_tokens: int = 4_000,    # 预留给模型输出的 token
        summary_threshold: float = 0.7,  # 当使用率超过 70% 时触发压缩
    ):
        self.model = model
        self.max_tokens = max_tokens
        self.reserve_tokens = reserve_tokens
        self.summary_threshold = summary_threshold
        self.encoder = tiktoken.encoding_for_model(model)

        self.system_message: Optional[dict] = None
        self.summary: Optional[str] = None           # 历史摘要
        self.messages: list[dict] = []                # 当前活跃消息

    def count_tokens(self, messages: list[dict]) -> int:
        """估算消息列表的 token 数"""
        total = 0
        for msg in messages:
            # 每条消息有固定开销（role + 分隔符）约 4 token
            total += 4
            content = msg.get("content", "")
            if content:
                total += len(self.encoder.encode(content))
        return total

    def set_system(self, content: str):
        """设置系统消息"""
        self.system_message = {"role": "system", "content": content}

    def add_message(self, role: str, content: str):
        """添加新消息，必要时触发压缩"""
        self.messages.append({"role": role, "content": content})

        # 检查是否需要压缩
        all_msgs = self._build_full_messages()
        used = self.count_tokens(all_msgs)
        available = self.max_tokens - self.reserve_tokens

        if used / available > self.summary_threshold:
            self._compress()

    def _compress(self):
        """将旧消息压缩为摘要"""
        if len(self.messages) <= 4:
            # 消息太少，无需压缩
            return

        # 保留最近 2 轮（4 条消息），压缩其余
        to_compress = self.messages[:-4]
        to_keep = self.messages[-4:]

        # 生成摘要
        compress_prompt = (
            "请将以下对话历史压缩为简洁的摘要，保留关键信息、决策和结果：\n\n"
        )
        if self.summary:
            compress_prompt += f"之前的摘要：{self.summary}\n\n"
        compress_prompt += "需要压缩的对话：\n"
        for msg in to_compress:
            compress_prompt += f"[{msg['role']}]: {msg['content'][:500]}\n"

        response = client.chat.completions.create(
            model=self.model,
            messages=[{"role": "user", "content": compress_prompt}],
            max_tokens=500,
            temperature=0,
        )
        self.summary = response.choices[0].message.content
        self.messages = to_keep
        print(f"  [上下文管理] 已压缩 {len(to_compress)} 条消息为摘要")

    def _build_full_messages(self) -> list[dict]:
        """构建完整的消息列表"""
        result = []
        if self.system_message:
            result.append(self.system_message)
        if self.summary:
            result.append({
                "role": "system",
                "content": f"[历史摘要] {self.summary}"
            })
        result.extend(self.messages)
        return result

    def get_messages(self) -> list[dict]:
        """获取当前完整消息列表，用于 API 调用"""
        return self._build_full_messages()

    def get_stats(self) -> dict:
        """获取当前上下文统计"""
        msgs = self._build_full_messages()
        used = self.count_tokens(msgs)
        return {
            "total_messages": len(msgs),
            "active_messages": len(self.messages),
            "has_summary": self.summary is not None,
            "tokens_used": used,
            "tokens_available": self.max_tokens - self.reserve_tokens,
            "usage_ratio": used / (self.max_tokens - self.reserve_tokens),
        }


# ── 使用示例 ──────────────────────────────────────────────
if __name__ == "__main__":
    mgr = ContextWindowManager(max_tokens=8000, summary_threshold=0.5)
    mgr.set_system("你是一个数据分析助手。")

    # 模拟多轮对话
    conversations = [
        ("user", "帮我分析上个月的销售数据"),
        ("assistant", "好的，我需要先查询数据库获取上个月的销售数据..."),
        ("user", "重点看华东地区"),
        ("assistant", "华东地区上月销售额 320 万，环比增长 15%..."),
        ("user", "和华南对比一下"),
        ("assistant", "华南地区上月销售额 280 万，环比增长 8%。华东领先华南 14%..."),
        ("user", "生成一份对比报告"),
        ("assistant", "已生成对比报告，主要结论：华东地区在增速和总量上均领先..."),
    ]

    for role, content in conversations:
        mgr.add_message(role, content)
        stats = mgr.get_stats()
        print(f"  消息数: {stats['active_messages']}, "
              f"token 使用率: {stats['usage_ratio']:.1%}, "
              f"有摘要: {stats['has_summary']}")
```

### 3.2 基于向量存储的长期记忆

```python
"""
基于向量存储的长期记忆系统
使用 chromadb 作为向量数据库（轻量级，适合本地开发）
"""
# pip install chromadb openai

import json
import time
from datetime import datetime
from openai import OpenAI
import chromadb

client = OpenAI()

class LongTermMemory:
    """
    Agent 的长期记忆系统。
    支持：存储记忆、语义检索、记忆衰减、去重。
    """

    def __init__(self, collection_name: str = "agent_memory"):
        self.chroma = chromadb.Client()  # 内存模式；生产环境用 PersistentClient
        self.collection = self.chroma.get_or_create_collection(
            name=collection_name,
            metadata={"hnsw:space": "cosine"},  # 余弦相似度
        )
        self.openai = OpenAI()

    def _embed(self, text: str) -> list[float]:
        """获取文本的 embedding 向量"""
        response = self.openai.embeddings.create(
            model="text-embedding-3-small",
            input=text,
        )
        return response.data[0].embedding

    def store(
        self,
        content: str,
        memory_type: str = "experience",   # experience / fact / preference
        importance: float = 0.5,            # 0~1 重要性
        metadata: dict = None,
    ) -> str:
        """存储一条记忆"""
        memory_id = f"mem_{int(time.time() * 1000)}"
        meta = {
            "type": memory_type,
            "importance": importance,
            "created_at": datetime.now().isoformat(),
            "access_count": 0,
        }
        if metadata:
            meta.update(metadata)

        self.collection.add(
            ids=[memory_id],
            documents=[content],
            embeddings=[self._embed(content)],
            metadatas=[meta],
        )
        return memory_id

    def retrieve(
        self,
        query: str,
        top_k: int = 5,
        memory_type: str = None,
        min_importance: float = 0.0,
    ) -> list[dict]:
        """语义检索相关记忆"""
        where_filter = {}
        if memory_type:
            where_filter["type"] = memory_type
        # chromadb 的 where 过滤
        if min_importance > 0:
            where_filter["importance"] = {"$gte": min_importance}

        results = self.collection.query(
            query_embeddings=[self._embed(query)],
            n_results=top_k,
            where=where_filter if where_filter else None,
        )

        memories = []
        if results["documents"] and results["documents"][0]:
            for i, doc in enumerate(results["documents"][0]):
                memories.append({
                    "id": results["ids"][0][i],
                    "content": doc,
                    "metadata": results["metadatas"][0][i],
                    "distance": results["distances"][0][i] if results.get("distances") else None,
                })
        return memories

    def format_for_context(self, memories: list[dict], max_tokens: int = 1000) -> str:
        """将检索到的记忆格式化为可注入上下文的文本"""
        if not memories:
            return ""
        lines = ["[相关记忆]"]
        for mem in memories:
            line = f"- [{mem['metadata'].get('type', 'unknown')}] {mem['content']}"
            lines.append(line)
        return "\n".join(lines)


# ── 使用示例 ──────────────────────────────────────────────
def demo_long_term_memory():
    memory = LongTermMemory("demo_agent")

    # 存储多条记忆
    memory.store(
        "用户偏好使用 Python，不喜欢 Java",
        memory_type="preference",
        importance=0.8,
    )
    memory.store(
        "上次帮用户部署应用时，Nginx 配置出了问题，最终发现是端口冲突",
        memory_type="experience",
        importance=0.7,
    )
    memory.store(
        "用户的项目使用 PostgreSQL 数据库，部署在 AWS 上",
        memory_type="fact",
        importance=0.9,
    )
    memory.store(
        "用户要求代码注释使用中文",
        memory_type="preference",
        importance=0.6,
    )

    # 语义检索
    query = "帮我写一个数据库查询脚本"
    results = memory.retrieve(query, top_k=3)
    context = memory.format_for_context(results)

    print(f"查询：{query}")
    print(f"检索到的记忆：\n{context}")
    # 这段 context 会被注入到 Agent 的 system prompt 中
```

### 3.3 Scratch Pad 工作记忆

```python
"""
Scratch Pad 工作记忆：Agent 执行复杂任务时的中间状态管理
"""
import json
from dataclasses import dataclass, field, asdict
from enum import Enum
from typing import Any

class StepStatus(str, Enum):
    PENDING = "pending"
    IN_PROGRESS = "in_progress"
    COMPLETED = "completed"
    FAILED = "failed"
    SKIPPED = "skipped"

@dataclass
class PlanStep:
    id: int
    description: str
    status: StepStatus = StepStatus.PENDING
    result: str = ""
    dependencies: list[int] = field(default_factory=list)

@dataclass
class ScratchPad:
    """
    Agent 的工作记忆 / 草稿纸。
    在每次 LLM 调用时，将当前状态注入到 prompt 中。
    """
    goal: str = ""
    plan: list[PlanStep] = field(default_factory=list)
    current_step_id: int = 0
    variables: dict[str, Any] = field(default_factory=dict)    # 中间变量
    notes: list[str] = field(default_factory=list)              # 随手记录
    error_log: list[str] = field(default_factory=list)          # 错误记录

    def set_goal(self, goal: str):
        self.goal = goal

    def add_step(self, description: str, dependencies: list[int] = None) -> int:
        step_id = len(self.plan) + 1
        self.plan.append(PlanStep(
            id=step_id,
            description=description,
            dependencies=dependencies or [],
        ))
        return step_id

    def update_step(self, step_id: int, status: StepStatus, result: str = ""):
        for step in self.plan:
            if step.id == step_id:
                step.status = status
                step.result = result
                return
        raise ValueError(f"Step {step_id} not found")

    def set_variable(self, key: str, value: Any):
        self.variables[key] = value

    def add_note(self, note: str):
        self.notes.append(note)

    def log_error(self, error: str):
        self.error_log.append(error)

    def get_next_step(self) -> PlanStep | None:
        """获取下一个可执行的步骤（所有依赖已完成）"""
        completed_ids = {s.id for s in self.plan if s.status == StepStatus.COMPLETED}
        for step in self.plan:
            if step.status == StepStatus.PENDING:
                if all(dep in completed_ids for dep in step.dependencies):
                    return step
        return None

    def to_prompt_string(self) -> str:
        """将工作记忆序列化为可注入 prompt 的字符串"""
        lines = ["[当前工作状态]"]
        lines.append(f"目标：{self.goal}")
        lines.append("")

        lines.append("执行计划：")
        status_icon = {
            StepStatus.PENDING: "○",
            StepStatus.IN_PROGRESS: "◎",
            StepStatus.COMPLETED: "●",
            StepStatus.FAILED: "✗",
            StepStatus.SKIPPED: "—",
        }
        for step in self.plan:
            icon = status_icon[step.status]
            line = f"  {icon} 步骤 {step.id}: {step.description}"
            if step.result:
                line += f" → {step.result[:100]}"
            lines.append(line)

        if self.variables:
            lines.append("")
            lines.append("中间变量：")
            for k, v in self.variables.items():
                lines.append(f"  {k} = {json.dumps(v, ensure_ascii=False)[:200]}")

        if self.notes:
            lines.append("")
            lines.append("备注：")
            for note in self.notes:
                lines.append(f"  - {note}")

        if self.error_log:
            lines.append("")
            lines.append("错误记录：")
            for err in self.error_log[-3:]:  # 只保留最近 3 条
                lines.append(f"  ! {err}")

        return "\n".join(lines)


# ── 使用示例：集成到 Agent 循环中 ─────────────────────────
def demo_scratchpad_agent():
    pad = ScratchPad()
    pad.set_goal("分析公司 Q1 财务数据并生成报告")

    # Agent 规划阶段生成步骤
    pad.add_step("从数据库获取 Q1 收入数据")
    pad.add_step("从数据库获取 Q1 支出数据")
    pad.add_step("计算利润率和同比增长", dependencies=[1, 2])
    pad.add_step("生成可视化图表", dependencies=[3])
    pad.add_step("撰写分析报告", dependencies=[3, 4])

    # 模拟执行过程
    pad.update_step(1, StepStatus.COMPLETED, "Q1 总收入 1200 万")
    pad.set_variable("q1_revenue", 12_000_000)

    pad.update_step(2, StepStatus.COMPLETED, "Q1 总支出 800 万")
    pad.set_variable("q1_expense", 8_000_000)

    pad.update_step(3, StepStatus.IN_PROGRESS)
    pad.set_variable("profit", 4_000_000)
    pad.add_note("需要获取去年 Q1 数据来计算同比增长")

    # 打印当前工作记忆（这段文本会被注入到 LLM 的 prompt 中）
    print(pad.to_prompt_string())
    print()

    # 获取下一步
    next_step = pad.get_next_step()
    if next_step:
        print(f"下一个可执行步骤：{next_step.description}")

if __name__ == "__main__":
    demo_scratchpad_agent()
```

## 4. 工程视角

### 4.1 记忆策略选型指南

```
你的 Agent 需要什么？

├── 只处理单轮/短对话（< 10 轮）
│   └── 不需要额外记忆管理，直接用上下文窗口
│
├── 长对话（10-100 轮）
│   └── 滑动窗口 + 摘要压缩
│       关键决策：摘要的粒度和频率
│
├── 跨会话记住用户偏好和历史
│   └── 向量存储（长期记忆）
│       关键决策：什么信息值得存储？如何去重？
│
├── 复杂多步任务（如数据分析、代码生成）
│   └── Scratch Pad（工作记忆）
│       关键决策：状态结构设计、何时更新
│
└── 以上都需要
    └── 三层记忆系统全部配备
        关键决策：各层之间如何协调、总 token 预算如何分配
```

### 4.2 Token 预算分配

一个典型的 128K 上下文窗口 Agent 的 token 预算分配：

```
┌────────────────────────────────────────────┐
│              128K Token 预算               │
├──────────────┬─────────────────────────────┤
│ System Prompt│ ~3K (2%)                    │
│ + 工具定义   │                             │
├──────────────┼─────────────────────────────┤
│ 长期记忆     │ ~5K (4%)  检索的 Top-K 记忆 │
│ (检索注入)   │                             │
├──────────────┼─────────────────────────────┤
│ 工作记忆     │ ~2K (2%)  当前任务状态      │
│ (Scratch Pad)│                             │
├──────────────┼─────────────────────────────┤
│ 历史摘要     │ ~3K (2%)  压缩的旧对话      │
├──────────────┼─────────────────────────────┤
│ 活跃对话     │ ~80K (63%) 最近的完整对话   │
├──────────────┼─────────────────────────────┤
│ 工具返回内容 │ ~25K (20%) 工具调用的结果   │
├──────────────┼─────────────────────────────┤
│ 模型输出预留 │ ~10K (8%)                   │
└──────────────┴─────────────────────────────┘
```

### 4.3 长期记忆的工程挑战

| 挑战 | 说明 | 应对方案 |
|------|------|----------|
| **信息过载** | 存储太多低质量记忆，检索效果下降 | 设置重要性阈值，定期清理 |
| **记忆冲突** | 新旧记忆矛盾（如用户改了偏好） | 时间戳优先，新记忆覆盖旧记忆 |
| **检索噪声** | 语义相似但不相关的记忆被召回 | 混合检索（向量 + 关键词），添加元数据过滤 |
| **隐私问题** | 跨会话记忆可能泄露敏感信息 | 加密存储、用户可控的记忆管理 |
| **存储成本** | 大量 embedding 向量的存储和计算 | 按需 embedding、定期归档 |

### 4.4 2025-2026 记忆技术进展

**模型原生记忆**：
- **ChatGPT Memory**（OpenAI, 2024）：模型自动从对话中提取并存储用户偏好，下次对话自动加载。是产品级长期记忆的标杆实现。
- **Claude Projects**（Anthropic, 2024-2025）：通过 Project Knowledge 实现项目级持久化上下文。

**框架层记忆**：
- **LangGraph Checkpointing**（2024-2025）：基于状态图的检查点机制，支持会话暂停和恢复。
- **Mem0**（2025）：开源的 Agent 记忆层，提供统一的长期记忆 API，支持多种后端。

**研究前沿**：
- **MemGPT / Letta**（2024-2025）：将操作系统的虚拟内存概念引入 LLM，自动在"主存"（上下文窗口）和"外存"（数据库）之间换页。
- **Retrieval-augmented generation + Memory**：将 RAG 和 Agent Memory 融合，让检索不仅覆盖外部知识库，也覆盖 Agent 的历史经验。

### 4.5 记忆系统的评测

| 评测维度 | 指标 | 说明 |
|----------|------|------|
| **检索准确率** | Recall@K, Precision@K | 检索的记忆是否相关 |
| **时效性** | 新鲜度分数 | 是否能优先返回最新信息 |
| **一致性** | 矛盾检测率 | 记忆之间是否存在冲突 |
| **效率** | 检索延迟、token 开销 | 记忆系统不应成为瓶颈 |
| **任务贡献** | 有/无记忆的任务完成率对比 | 记忆是否真的有帮助 |

## 时效性说明

本节内容截至 2026 年 4 月。以下领域正在快速变化：
- **上下文窗口持续扩大**：Gemini 已支持 2M token，更大窗口可能减少对外部记忆的需求，但不会消除它——因为成本和延迟与窗口大小成正比
- **模型原生记忆能力**：各厂商正在将记忆管理内置到模型/产品层，框架层的记忆方案可能被逐步替代
- **向量数据库生态**：Pinecone、Weaviate、Qdrant、Chroma 等产品迭代迅速，API 和最佳实践以官方文档为准
