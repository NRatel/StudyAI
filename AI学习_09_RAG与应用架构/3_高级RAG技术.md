# 高级 RAG 技术

> **前置知识**：[Embedding 与向量检索](2_Embedding与向量检索.md)
>
> **本节目标**：理解 RAG 的优化重点不是堆组件，而是让证据更准、上下文更干净、答案更可验证。

---

## 1. RAG 的核心不是“接个向量库”

很多 RAG 系统效果不好，不是因为没有向量数据库，而是因为检索和上下文质量不够。

RAG 真正要解决的是：

> 找到对问题有用、可信、完整、不过量的证据。

## 2. 混合检索：语义和关键词都要

向量检索擅长找意思相近的内容。关键词检索擅长找精确词、编号、术语、代码、日期。

混合检索把两者结合起来。

比如查“错误码 E1024”，关键词比纯语义更可靠；查“如何恢复账号访问”，语义检索更灵活。

## 3. Query Rewrite：先把问题改写清楚

用户的问题可能很短、含糊、带上下文。

系统可以先把问题改写成更适合检索的形式。

比如用户问“这个怎么续费”，系统需要根据对话上下文改成“某产品会员如何续费”。

## 4. Rerank：把真正相关的排前面

第一轮检索通常召回一批候选，里面会混着不太相关的内容。

Rerank 会更仔细地判断问题和文档片段是否匹配，把最有用的片段排前面。

这一步常常比单纯调 top-k 更有效。

## 5. 评估：不能只看回答顺不顺

RAG 评估要分开看：

- 检索有没有找到正确证据。
- 排序有没有把关键证据放前面。
- 生成有没有忠实使用证据。
- 答案有没有引用来源。
- 没有证据时是否承认不知道。

一个回答写得流畅，不代表 RAG 成功。关键是它是否基于正确证据。
