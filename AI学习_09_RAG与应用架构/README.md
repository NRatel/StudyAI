# AI学习：09 RAG与应用架构

> 读本模块时，先抓住大模型为什么需要外部知识，以及 RAG 的检索、重排、生成、评估分别在解决什么问题。不需要直接搭建生产级知识库。

## 本模块导读

**学什么**：检索增强生成（Retrieval-Augmented Generation, RAG）——当前将大语言模型接入外部知识的核心架构。本模块从"为什么需要 RAG"出发，贯穿索引构建、向量检索、高级检索策略，直到完整 RAG 系统的设计思路。

**RAG 与长上下文的关系**：RAG 与长上下文窗口不是替代关系，而是互补关系。RAG 负责从海量数据（可能是百万篇文档）中**检索**出少量相关内容，长上下文负责在这些检索结果之间进行**推理和综合**。即使模型支持 128K 甚至更长的上下文，也不可能把整个知识库塞进去——RAG 负责"找到"，长上下文负责"理解"，两者配合效果最好。

**为什么学**：大语言模型有三个根本性弱点——幻觉（编造不存在的事实）、时效性（训练数据有截止日期）、私有数据隔离（无法访问企业内部知识）。RAG 是工业界解决这三个问题的主流方案，也是当前 AI 应用最密集的工程方向。无论你要构建智能客服、企业知识库、代码助手还是研究工具，RAG 都是核心技术栈。

**读完能看懂的主线**：
- 理解 RAG 的完整流程：文档索引构建 → 语义检索 → 增强生成
- 理解 Embedding、向量数据库、Chunking 策略分别影响什么
- 能解释 Query 改写、HyDE、Reranker、混合检索等高级 RAG 技术解决的问题
- 理解 RAGAS 等评估框架关注哪些质量维度
- 理解 RAG 系统的主要设计取舍，以及 Agentic RAG 的演进方向

## 读完能解释

1. RAG 为什么能缓解幻觉、时效性和私有知识问题？

3. Chunking 为什么不是简单“按字数切块”？切得太大或太小各有什么问题？

## 阅读建议

按序号 1→3 顺序阅读，每个文件都会从具体问题切入，再讲清关键机制，最后收束关键疑问。

- **第 1 节**是全局地图——先建立对 RAG 全流程的整体认知
- **第 2 节**深入"检索"这一核心环节——Embedding、向量数据库、Chunking 是 RAG 效果的基石
- **第 3 节**是进阶——当 Naive RAG 效果不够时，如何通过各种高级技术持续提升

建议先把全流程概念跑通：如果需要验证直觉，可以用一个最小本地知识库问答系统观察检索、重排、生成各环节如何影响结果。
