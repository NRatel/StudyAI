## 一、直觉：从链到树到图

### 推理结构的进化

上一节的 CoT 是一条**线性链**：一步接一步，从问题走向答案。

但人类解决复杂问题时，往往不是一条线走到底的：
- **尝试多条路**：数学竞赛选手会尝试不同的解题思路，选最有希望的继续深入
- **回头修正**：发现走不通时，会退回来换一条路
- **反思改进**：做完后回头看，思考哪里可以做得更好

这些直觉催生了三个方向的进化：

```
Chain-of-Thought (CoT)      一条直线
    │
    ├── Tree-of-Thought      一棵搜索树（多路径 + 剪枝）
    │
    ├── Graph-of-Thought     一张图（节点可合并、可回溯）
    │
    └── Reflexion            自我反思循环（做完回头看 + 改进）

另外：ReAct = 推理 + 行动交替（与外部工具交互）
```

## 二、Tree-of-Thought (ToT)

### 2.1 核心思想（Yao et al., 2023）

**论文**：*Tree of Thoughts: Deliberate Problem Solving with Large Language Models*（Yao et al., NeurIPS 2023）

CoT 的局限在于：一旦某一步走错了，后续所有步骤都会沿着错误方向继续。Self-Consistency 通过"重新开始"来缓解，但每次都是从头重来。

ToT 的核心创新是把推理过程从**链式结构**变成**树形搜索**：

```
              [问题]
             /  |  \
          思路A 思路B 思路C      ← 第 1 步：生成多个候选思路
         / \    |    / \
       A1  A2   B1  C1  C2     ← 第 2 步：对每个思路继续展开
       ↓        ↓              ← 评估并剪枝：只保留有希望的
      A1'       B1'
       ↓        ↓              ← 继续推理
     [答案1]  [答案2]          ← 选择最佳答案
```

**三个关键组件**：

1. **Thought Decomposition（思维分解）**：将推理过程分成若干"思维步骤"，每步生成多个候选
2. **Thought Evaluation（思维评估）**：用 LLM 自身评估每个候选的质量（"这条路有希望吗？"）
3. **Search Algorithm（搜索算法）**：BFS（广度优先）或 DFS（深度优先）来遍历思维树

### 2.2 与传统搜索的对比

| 组件 | 传统搜索（如 A*） | Tree-of-Thought |
|------|-------------------|-----------------|
| 状态 | 棋盘、迷宫位置等 | 推理的中间文本 |
| 动作 | 移动、落子等 | 生成下一步推理 |
| 评估函数 | 手工设计的启发函数 | LLM 自我评估（"这一步好不好？"） |
| 搜索策略 | BFS / DFS / A* | BFS / DFS（由 LLM 驱动） |

### 2.3 经典案例：24 点游戏

**任务**：给定 4 个数字，用 +−×÷ 和括号组合得到 24。

```
输入: 1, 2, 3, 4

标准 CoT: 模型一次性生成一个表达式，成功率约 7.3%

ToT 流程:
第 1 步 — 生成候选操作（选两个数做运算）:
  候选 A: 1 + 2 = 3    (剩余: 3, 3, 4)
  候选 B: 3 × 4 = 12   (剩余: 1, 2, 12)
  候选 C: 2 × 3 = 6    (剩余: 1, 4, 6)
  候选 D: 1 × 4 = 4    (剩余: 2, 3, 4)

第 1 步 — 评估:
  候选 A: "3, 3, 4 → 可以 3 × (4+3)=21? 不太容易到 24" → 评分: 低
  候选 B: "1, 2, 12 → 12 × 2 = 24!" → 评分: 高 ✓
  候选 C: "1, 4, 6 → 4 × 6 = 24!" → 评分: 高 ✓
  候选 D: "2, 3, 4 → 2 × 3 × 4 = 24!" → 评分: 高 ✓

第 2 步 — 对高分候选继续展开...
```

**结果**：ToT 在 24 点任务上成功率 74%，远超 CoT 的 7.3%。

### 2.4 Python 实现

```python
"""
Tree-of-Thought 简化实现
以 24 点游戏为例
"""
from openai import OpenAI
import re

client = OpenAI()

def generate_next_steps(state: str, n_candidates: int = 5) -> list[str]:
    """
    生成下一步的多个候选方案
    state: 当前状态描述（如 "剩余数字: 1, 2, 12"）
    """
    prompt = f"""
你在玩 24 点游戏。当前状态：{state}
请生成 {n_candidates} 种不同的下一步操作。
每种操作选取两个数进行一次四则运算。
格式：
操作: X op Y = Z (剩余: ...)
"""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.8
    )
    text = response.choices[0].message.content
    # 解析每个候选操作
    steps = [line.strip() for line in text.split('\n') if line.strip().startswith('操作')]
    return steps[:n_candidates]

def evaluate_step(state: str, step: str) -> float:
    """
    用 LLM 评估某一步的质量
    返回 0~1 的评分：这一步离达到 24 有多大希望？
    """
    prompt = f"""
24 点游戏。当前状态：{state}
执行操作：{step}
请评估这步操作后，剩余数字最终能否组合出 24。
回答 "确定可以"、"可能可以" 或 "不太可能"。
"""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        temperature=0
    )
    text = response.choices[0].message.content.lower()
    if "确定可以" in text or "sure" in text:
        return 1.0
    elif "可能可以" in text or "maybe" in text or "likely" in text:
        return 0.5
    else:
        return 0.1

def tree_of_thought_bfs(
    numbers: list[int],
    max_depth: int = 3,
    beam_width: int = 3
) -> str | None:
    """
    BFS 版 Tree-of-Thought

    Args:
        numbers: 初始数字列表
        max_depth: 最大搜索深度（每步做一次运算）
        beam_width: 每步保留的最佳候选数

    Returns:
        成功的表达式路径，或 None
    """
    # 初始状态
    initial_state = f"数字: {', '.join(map(str, numbers))}，目标: 24"
    current_beam = [(initial_state, [])]  # (状态, 操作历史)

    for depth in range(max_depth):
        candidates = []

        for state, history in current_beam:
            # 生成下一步候选
            next_steps = generate_next_steps(state)

            for step in next_steps:
                # 评估每个候选
                score = evaluate_step(state, step)
                candidates.append((step, state, history + [step], score))

        # 按评分排序，保留 beam_width 个最佳候选
        candidates.sort(key=lambda x: x[3], reverse=True)
        current_beam = []

        for step, prev_state, history, score in candidates[:beam_width]:
            # 检查是否已得到 24
            if "= 24" in step or "得到 24" in step:
                return "\n".join(history)
            # 更新状态（简化处理）
            new_state = f"上一步: {step}"
            current_beam.append((new_state, history))

    return None

# ----- 使用示例 -----
result = tree_of_thought_bfs([1, 2, 3, 4])
if result:
    print(f"找到解法:\n{result}")
else:
    print("未找到解法")
```

### 2.5 ToT 的局限

| 局限 | 说明 |
|------|------|
| API 调用次数多 | 每个节点需要"生成"和"评估"两次 LLM 调用，树越大越贵 |
| LLM 评估不准 | 模型评估"有没有希望"有时不靠谱，导致剪掉正确路径 |
| 任务适应性 | 效果在结构化问题（24 点、填字）上最好，开放式问题不太适用 |
| 延迟高 | 多轮搜索导致端到端延迟很高，不适合实时交互 |

## 三、Graph-of-Thought (GoT)

### 3.1 核心思想（Besta et al., 2023）

**论文**：*Graph of Thoughts: Solving Elaborate Problems with Large Language Models*（Besta et al., 2023）

ToT 是树形结构，有一个限制：**不同分支的信息不能合并**。但在实际推理中，两条思路的部分结论可以互相补充。

GoT 将推理结构从树扩展为**有向图**：

```
Tree (ToT):                    Graph (GoT):
    A                              A ←──┐
   / \                            / \    │
  B   C                          B   C   │ (合并/反馈)
 / \   \                        / \ / \  │
D   E   F                      D  E+F  G─┘

树：分支独立，不能合并         图：分支可合并，可回环
```

**GoT 新增的操作**：

| 操作 | 说明 | 示例 |
|------|------|------|
| **聚合（Aggregate）** | 合并多个思维节点的信息 | 把路径 A 和路径 B 的部分结论合并 |
| **精炼（Refine）** | 在已有节点上迭代改进 | 基于新信息修正之前的结论 |
| **回溯（Backtrack）** | 返回之前的节点重新开始 | 发现当前路径不对，退回 |

### 3.2 适用场景

GoT 在以下场景比 ToT 更有优势：
- **排序问题**：可以把数组分成多块分别排序，再合并
- **文本摘要**：可以对不同段落分别摘要，再聚合成最终摘要
- **代码审查**：分别分析不同模块的问题，再综合给出整体评估

### 3.3 简化实现思路

```python
"""
Graph-of-Thought 核心概念实现
展示 GoT 的图结构和操作
"""
from dataclasses import dataclass, field
from openai import OpenAI

client = OpenAI()

@dataclass
class ThoughtNode:
    """思维节点"""
    id: str
    content: str                         # 当前思维内容
    parents: list[str] = field(default_factory=list)  # 父节点 ID
    children: list[str] = field(default_factory=list)  # 子节点 ID
    score: float = 0.0                   # 评估分数

class GraphOfThought:
    """Graph-of-Thought 推理引擎"""

    def __init__(self, model="gpt-4o"):
        self.model = model
        self.nodes: dict[str, ThoughtNode] = {}
        self.node_counter = 0

    def add_thought(self, content: str, parents: list[str] = None) -> str:
        """添加思维节点"""
        node_id = f"T{self.node_counter}"
        self.node_counter += 1
        parents = parents or []
        node = ThoughtNode(id=node_id, content=content, parents=parents)
        self.nodes[node_id] = node
        for pid in parents:
            if pid in self.nodes:
                self.nodes[pid].children.append(node_id)
        return node_id

    def generate(self, parent_id: str, prompt: str, n: int = 3) -> list[str]:
        """从一个节点生成多个子思维"""
        parent = self.nodes[parent_id]
        response = client.chat.completions.create(
            model=self.model,
            messages=[{
                "role": "user",
                "content": f"基于以下思考：\n{parent.content}\n\n{prompt}\n请给出 {n} 种不同的思路。"
            }],
            temperature=0.8
        )
        text = response.choices[0].message.content
        thoughts = [t.strip() for t in text.split('\n\n') if t.strip()][:n]
        return [self.add_thought(t, [parent_id]) for t in thoughts]

    def aggregate(self, node_ids: list[str], prompt: str) -> str:
        """
        聚合操作：合并多个思维节点
        这是 GoT 相比 ToT 的核心新操作
        """
        contents = [f"[思路 {nid}]: {self.nodes[nid].content}" for nid in node_ids]
        combined = "\n".join(contents)
        response = client.chat.completions.create(
            model=self.model,
            messages=[{
                "role": "user",
                "content": f"以下是多条推理思路：\n{combined}\n\n{prompt}\n请综合以上思路，给出更好的结论。"
            }],
            temperature=0
        )
        merged_content = response.choices[0].message.content
        return self.add_thought(merged_content, node_ids)

    def refine(self, node_id: str, feedback: str) -> str:
        """精炼操作：基于反馈改进某个思维"""
        node = self.nodes[node_id]
        response = client.chat.completions.create(
            model=self.model,
            messages=[{
                "role": "user",
                "content": f"原始思考：\n{node.content}\n\n反馈：{feedback}\n\n请基于反馈改进你的思考。"
            }],
            temperature=0
        )
        refined_content = response.choices[0].message.content
        return self.add_thought(refined_content, [node_id])

# ----- 使用示例：文本摘要的 GoT -----
got = GraphOfThought()

# 第 1 步：分别对长文本的不同部分做摘要
root = got.add_thought("原文：[一篇很长的技术报告...]")
part1 = got.add_thought("第一部分摘要：提出了新的注意力机制...", [root])
part2 = got.add_thought("第二部分摘要：实验显示在多个基准上提升...", [root])
part3 = got.add_thought("第三部分摘要：消融实验证明关键组件...", [root])

# 第 2 步：聚合（GoT 的核心操作）
merged = got.aggregate(
    [part1, part2, part3],
    "综合三部分摘要，生成一段完整的论文摘要"
)

# 第 3 步：精炼
final = got.refine(merged, "请更加简洁，控制在 100 字以内")
print(got.nodes[final].content)
```

## 四、Self-Reflection / Reflexion

### 4.1 核心思想（Shinn et al., 2023）

**论文**：*Reflexion: Language Agents with Verbal Reinforcement Learning*（Shinn et al., NeurIPS 2023）

CoT、ToT、GoT 都是在"同一次"推理中做优化。Reflexion 引入了一个新维度：**跨轮次的自我反思**。

```
Reflexion 循环:

轮次 1: 尝试解题 → 得到答案 → 检查：错了！
         ↓
         反思："我在第 3 步犯了计算错误，把乘法写成了加法"
         ↓
轮次 2: 重新解题（带着上轮的反思） → 得到答案 → 检查：对了！
```

**核心组件**：

1. **Actor（执行者）**：LLM，负责实际执行任务（写代码、解题等）
2. **Evaluator（评估者）**：判断结果是否正确（可以是测试用例、另一个 LLM、或人工反馈）
3. **Self-Reflection（自我反思）**：LLM 分析失败原因，生成文字形式的"经验教训"
4. **Memory（记忆）**：存储历史反思，在下一轮注入上下文

### 4.2 与人类学习的类比

| Reflexion 组件 | 人类类比 |
|----------------|----------|
| Actor | 考试时答题 |
| Evaluator | 老师批改 |
| Self-Reflection | 分析错题，写错题笔记 |
| Memory | 错题本，下次考试前翻看 |

### 4.3 Reflexion 在代码生成中的应用

Reflexion 在代码生成任务中效果尤其显著（HumanEval: 80.1% → 91.0%），因为代码有天然的评估器——单元测试。

```python
"""
Reflexion 在代码生成中的实现
"""
from openai import OpenAI
import subprocess
import tempfile
import os

client = OpenAI()

def generate_code(task: str, reflections: list[str] = None) -> str:
    """让 LLM 生成代码，可选地注入历史反思"""
    reflection_text = ""
    if reflections:
        reflection_text = "\n\n### 之前的尝试和反思：\n"
        for i, r in enumerate(reflections, 1):
            reflection_text += f"\n第 {i} 次反思：{r}\n"

    prompt = f"""
请完成以下编程任务：
{task}
{reflection_text}
请只输出 Python 代码，不需要解释。
"""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        temperature=0
    )
    code = response.choices[0].message.content
    # 去除 markdown 代码块标记
    if "```python" in code:
        code = code.split("```python")[1].split("```")[0]
    elif "```" in code:
        code = code.split("```")[1].split("```")[0]
    return code.strip()

def evaluate_code(code: str, test_cases: str) -> tuple[bool, str]:
    """
    运行代码和测试用例
    返回 (是否通过, 错误信息)
    """
    full_code = code + "\n\n" + test_cases
    with tempfile.NamedTemporaryFile(mode='w', suffix='.py', delete=False) as f:
        f.write(full_code)
        f.flush()
        try:
            result = subprocess.run(
                ['python', f.name],
                capture_output=True, text=True, timeout=10
            )
            if result.returncode == 0:
                return True, "所有测试通过"
            else:
                return False, result.stderr
        except subprocess.TimeoutExpired:
            return False, "执行超时"
        finally:
            os.unlink(f.name)

def reflect_on_failure(task: str, code: str, error: str) -> str:
    """LLM 自我反思：分析失败原因"""
    prompt = f"""
你之前尝试完成以下任务但失败了：

任务：{task}

你写的代码：
```python
{code}
```

错误信息：
{error}

请分析失败原因，并总结你学到的教训。注意：
1. 具体指出哪一行/哪个逻辑出了问题
2. 说明正确的做法应该是什么
3. 不要重写代码，只做反思分析
"""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        temperature=0
    )
    return response.choices[0].message.content

def reflexion_loop(
    task: str,
    test_cases: str,
    max_trials: int = 3
) -> dict:
    """
    Reflexion 主循环

    Args:
        task: 编程任务描述
        test_cases: 测试用例代码
        max_trials: 最大尝试次数

    Returns:
        包含最终代码、是否成功、尝试次数的字典
    """
    reflections = []

    for trial in range(max_trials):
        print(f"\n--- 第 {trial + 1} 次尝试 ---")

        # 1. 生成代码（注入历史反思）
        code = generate_code(task, reflections if reflections else None)
        print(f"生成的代码:\n{code}")

        # 2. 评估
        passed, error_msg = evaluate_code(code, test_cases)
        print(f"评估结果: {'通过' if passed else '失败'}")

        if passed:
            return {
                "code": code,
                "success": True,
                "trials": trial + 1,
                "reflections": reflections
            }

        # 3. 反思
        reflection = reflect_on_failure(task, code, error_msg)
        reflections.append(reflection)
        print(f"反思: {reflection[:200]}...")

    return {
        "code": code,
        "success": False,
        "trials": max_trials,
        "reflections": reflections
    }

# ----- 使用示例 -----
task = """
实现一个函数 is_balanced(s)，检查字符串 s 中的括号是否平衡。
支持 ()、[]、{} 三种括号。
"""

test_cases = """
assert is_balanced("()") == True
assert is_balanced("([{}])") == True
assert is_balanced("(]") == False
assert is_balanced("([)]") == False
assert is_balanced("") == True
assert is_balanced("{[()]}") == True
assert is_balanced("((())") == False
print("所有测试通过！")
"""

result = reflexion_loop(task, test_cases, max_trials=3)
print(f"\n最终结果: {'成功' if result['success'] else '失败'}，用了 {result['trials']} 次尝试")
```

### 4.4 Reflexion 的关键设计

```
                  ┌──────────────┐
                  │   任务描述    │
                  └──────┬───────┘
                         │
                         ▼
              ┌─────────────────────┐
              │  Actor (LLM 执行)   │ ← 注入历史反思
              └──────────┬──────────┘
                         │ 输出
                         ▼
              ┌─────────────────────┐
              │ Evaluator (测试/评估)│
              └──────────┬──────────┘
                         │
                    通过? ──── 是 ──→ 完成！
                         │
                         否
                         │
                         ▼
              ┌─────────────────────┐
              │ Self-Reflection     │
              │ (分析失败原因)       │
              └──────────┬──────────┘
                         │ 文字形式的"经验"
                         ▼
              ┌─────────────────────┐
              │  Memory (记忆存储)   │ → 注入下一轮 Actor
              └─────────────────────┘
```

## 五、ReAct：推理 + 行动

### 5.1 核心思想（Yao et al., 2023）

**论文**：*ReAct: Synergizing Reasoning and Acting in Language Models*（Yao et al., ICLR 2023）

ReAct 将推理（Reasoning）和行动（Acting）交替进行：

```
传统 CoT:    Thought → Thought → Thought → Answer
             (纯思考，不与外界交互)

传统 Acting: Action → Observation → Action → Observation
             (纯行动，不进行推理)

ReAct:       Thought → Action → Observation → Thought → Action → ...
             (想一步、做一步、看结果、再想再做)
```

### 5.2 经典示例

```
Question: 科罗拉多造山运动东部的什么山脉的最高峰海拔多少？

Thought 1: 我需要搜索科罗拉多造山运动，了解其东部有什么山脉。
Action 1: Search["科罗拉多造山运动"]
Observation 1: 科罗拉多造山运动影响了落基山脉东部的前山脉带...

Thought 2: 前山脉（Front Range）在科罗拉多造山运动的东部。我需要找前山脉的最高峰。
Action 2: Search["Front Range 最高峰"]
Observation 2: 前山脉的最高峰是 Grays Peak，海拔 4,352 米。

Thought 3: 我找到答案了。
Action 3: Finish["4,352 米"]
```

### 5.3 实现框架

```python
"""
ReAct 模式的简化实现
展示 Thought/Action/Observation 循环
"""
from openai import OpenAI
import json

client = OpenAI()

# 定义可用工具
TOOLS = {
    "search": lambda query: f"[模拟搜索结果: {query} 的相关信息...]",
    "calculate": lambda expr: str(eval(expr)),  # 简化，实际应更安全
    "lookup": lambda term: f"[模拟查找: {term} 的详细信息...]"
}

REACT_SYSTEM_PROMPT = """
你是一个推理与行动交替进行的助手。对于每个步骤，你可以：

1. Thought: 思考当前情况，规划下一步
2. Action: 使用工具获取信息。可用工具：
   - search(query): 搜索信息
   - calculate(expression): 计算数学表达式
   - lookup(term): 查找特定术语
3. 当你确定答案时，使用 Action: finish(answer)

严格按以下格式输出（一次只输出一个 Thought + Action）：
Thought: [你的思考]
Action: [工具名](参数)
"""

def react_loop(question: str, max_steps: int = 5) -> str:
    """
    ReAct 推理循环

    交替进行：Thought → Action → Observation → Thought → ...
    """
    messages = [
        {"role": "system", "content": REACT_SYSTEM_PROMPT},
        {"role": "user", "content": f"Question: {question}"}
    ]

    for step in range(max_steps):
        # LLM 生成 Thought + Action
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            temperature=0,
            stop=["Observation:"]
        )
        assistant_msg = response.choices[0].message.content
        messages.append({"role": "assistant", "content": assistant_msg})

        print(f"\n=== Step {step + 1} ===")
        print(assistant_msg)

        # 解析 Action
        if "finish(" in assistant_msg.lower():
            import re
            match = re.search(r'finish\((.+?)\)', assistant_msg, re.IGNORECASE)
            if match:
                return match.group(1).strip('"\'')

        # 执行 Action 并获取 Observation
        # （简化的动作解析）
        import re
        action_match = re.search(r'Action:\s*(\w+)\((.+?)\)', assistant_msg)
        if action_match:
            tool_name = action_match.group(1).lower()
            tool_arg = action_match.group(2).strip('"\'')
            if tool_name in TOOLS:
                observation = TOOLS[tool_name](tool_arg)
            else:
                observation = f"未知工具: {tool_name}"
        else:
            observation = "无法解析动作，请重试。"

        # 将 Observation 加入上下文
        obs_msg = f"Observation: {observation}"
        messages.append({"role": "user", "content": obs_msg})
        print(obs_msg)

    return "达到最大步数，未能得出结论"

# ----- 使用示例 -----
answer = react_loop("2024 年诺贝尔物理学奖得主是谁？他们的主要贡献是什么？")
print(f"\n最终答案: {answer}")
```

### 5.4 ReAct 与 Agent 的关系

ReAct 是 AI Agent 架构的理论基石。如果你后续学习模块 12（Agent 与工具使用），会看到 LangChain、LangGraph 等框架的核心循环就是 ReAct 模式的工程化实现。

```
ReAct (论文层面)                    Agent 框架 (工程层面)
━━━━━━━━━━━━━━━━                  ━━━━━━━━━━━━━━━━━━━
Thought                      →     LLM 推理 / Planning
Action                       →     Tool Use / Function Calling
Observation                  →     工具返回结果
Memory (Reflexion 扩展)      →     Agent 记忆模块
```

## 六、方法对比与选择指南

### 6.1 综合对比

| 方法 | 结构 | LLM 调用次数 | 最适合的任务 | 典型提升 |
|------|------|-------------|-------------|----------|
| CoT | 链 | 1 次 | 中等推理 | +15~20% (GSM8K) |
| Self-Consistency | 多条链 | N 次(5~40) | 有标准答案的推理 | +10~18% (GSM8K) |
| ToT | 树 | O(b^d) 次 | 结构化搜索问题 | +67% (24 点) |
| GoT | 图 | 灵活 | 可分解后合并的问题 | 因任务而异 |
| Reflexion | 多轮循环 | 多轮 × 每轮若干 | 有评估器的任务 | +11% (HumanEval) |
| ReAct | 交替链 | 多步 | 需要外部信息的问题 | 因任务而异 |

### 6.2 选择决策树

```
你的推理任务是什么类型？
│
├── 有标准答案的推理（数学/逻辑）
│   ├── 预算充足 → Self-Consistency (5~10 次采样)
│   ├── 需要搜索式探索 → ToT
│   └── 预算有限 → Zero-shot CoT
│
├── 需要与外部工具交互
│   └── ReAct（推理 + 行动循环）
│
├── 可以多轮迭代改进
│   └── Reflexion（尤其适合代码生成）
│
├── 可分解为子任务再合并
│   └── GoT（分别处理 + 聚合）
│
└── 简单直接的问题
    └── 不需要高级推理方法，直接回答
```

## 七、工程视角：高级推理方法的实际应用

### 7.1 工程化建议

```python
"""
生产环境中选择推理策略的参考实现
"""

class AdaptiveReasoner:
    """
    自适应推理引擎

    根据问题的复杂度自动选择推理策略：
    - 简单问题 → 直接回答
    - 中等问题 → Zero-shot CoT
    - 复杂问题 → Self-Consistency
    - 需要搜索 → ReAct

    在 2025-2026 年，如果使用推理模型（o1/o3），
    可以跳过这些策略，让模型自己处理。见第 3 节。
    """

    def __init__(self, client, model="gpt-4o"):
        self.client = client
        self.model = model

    def classify_complexity(self, question: str) -> str:
        """用 LLM 快速判断问题复杂度"""
        response = self.client.chat.completions.create(
            model=self.model,
            messages=[{
                "role": "user",
                "content": f"请判断以下问题的复杂度（简单/中等/复杂/需要搜索）：\n{question}\n只回答一个词。"
            }],
            temperature=0,
            max_tokens=10
        )
        return response.choices[0].message.content.strip()

    def solve(self, question: str, force_strategy: str = None) -> dict:
        """
        自适应求解

        实际生产中的考虑：
        1. 分类本身也有成本（一次 LLM 调用），简单场景可省略
        2. 可以用正则/规则初步分类，只对不确定的用 LLM 分类
        3. 错误的分类会导致过度/不足推理，需要监控和调优
        """
        strategy = force_strategy or self._auto_select(question)
        # 根据策略执行相应的推理方法...
        return {"strategy": strategy, "answer": "..."}

    def _auto_select(self, question: str) -> str:
        complexity = self.classify_complexity(question)
        mapping = {
            "简单": "direct",
            "中等": "zero_shot_cot",
            "复杂": "self_consistency",
            "需要搜索": "react"
        }
        return mapping.get(complexity, "zero_shot_cot")
```

### 7.2 常见陷阱

| 陷阱 | 说明 | 建议 |
|------|------|------|
| 过度工程 | 所有问题都用 ToT/GoT | 先测 CoT 基线，不够再升级 |
| 忽视延迟 | ToT 的多轮搜索可能需要 30s+ | 为用户设置合理预期，或用流式输出 |
| 评估器质量差 | ToT/Reflexion 的效果取决于评估质量 | 如果评估器不靠谱，整个方法都不靠谱 |
| 忽视成本 | Self-Consistency × 40 = 40 倍成本 | 5~10 次通常够了，收益递减 |

## 八、从提示方法到训练方法的转折

到目前为止，我们讨论的所有方法都有一个共同特点：**不改变模型参数**。

CoT、ToT、GoT、Reflexion、ReAct 都是在"如何提问"和"如何组织推理过程"上做文章。模型本身并不知道自己在做"树形搜索"或"自我反思"——这些都是外部框架在调度。

2024 年起，另一条路线取得了突破性进展：**直接训练模型学会推理**。

```
提示方法 (2022-2023):                  训练方法 (2024-2026):
外部框架控制推理过程                    模型自身学会推理过程
- CoT: 提示模型写步骤                  - o1: 模型自动产生长推理链
- ToT: 外部框架做搜索                  - R1: RL 训练模型"慢思考"
- Reflexion: 外部框架做反思循环        - GRPO: 群组策略优化训练推理

优点: 通用，不需训练                    优点: 更强，更自然，更可靠
缺点: 依赖外部编排，开销大              缺点: 需要专门训练，成本高
```

下一节将深入探讨这条"训练推理模型"的路线。

## 本节小结

| 概念 | 一句话解释 |
|------|-----------|
| Tree-of-Thought | 把推理变成树形搜索：多路径展开 + LLM 评估 + 剪枝 |
| Graph-of-Thought | 进一步扩展为图结构：支持分支合并和迭代精炼 |
| Reflexion | 跨轮次的自我反思：失败后分析原因，下一轮改进 |
| ReAct | 推理和行动交替：想一步、做一步、看结果、再想再做 |
| 关键局限 | 这些方法都不改变模型参数，依赖外部框架编排 |

## 时效性说明

> - ToT、GoT 的原始论文发表于 2023 年，方法论已相对稳定
> - Reflexion 模式在 2024-2025 年被广泛集成到 Agent 框架中（LangGraph、CrewAI 等）

> - 但在开源小模型和成本敏感场景中，提示方法仍有实用价值
