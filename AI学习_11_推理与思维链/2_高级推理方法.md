## 一、直觉：从链到树到图

### 推理结构的进化

上一节的 CoT 是一条**线性链**：一步接一步，从问题走向答案。

但人类解决复杂问题时，往往不是一条线走到底的：
- **尝试多条路**：数学竞赛选手会尝试不同的解题思路，选最有希望的继续深入
- **回头修正**：发现走不通时，会退回来换一条路
- **反思改进**：做完后回头看，思考哪里可以做得更好

这些直觉催生了三个方向的进化：

```
Chain-of-Thought (CoT)      一条直线
    │
    ├── Tree-of-Thought      一棵搜索树（多路径 + 剪枝）
    │
    ├── Graph-of-Thought     一张图（节点可合并、可回溯）
    │
    └── Reflexion            自我反思循环（做完回头看 + 改进）

另外：ReAct = 推理 + 行动交替（与外部工具交互）
```

## 二、Tree-of-Thought (ToT)

### 2.1 核心思想（Yao et al., 2023）

**论文**：*Tree of Thoughts: Deliberate Problem Solving with Large Language Models*（Yao et al., NeurIPS 2023）

CoT 的局限在于：一旦某一步走错了，后续所有步骤都会沿着错误方向继续。Self-Consistency 通过"重新开始"来缓解，但每次都是从头重来。

ToT 的核心创新是把推理过程从**链式结构**变成**树形搜索**：

```
              [问题]
             /  |  \
          思路A 思路B 思路C      ← 第 1 步：生成多个候选思路
         / \    |    / \
       A1  A2   B1  C1  C2     ← 第 2 步：对每个思路继续展开
       ↓        ↓              ← 评估并剪枝：只保留有希望的
      A1'       B1'
       ↓        ↓              ← 继续推理
     [答案1]  [答案2]          ← 选择最佳答案
```

**三个关键组件**：

1. **Thought Decomposition（思维分解）**：将推理过程分成若干"思维步骤"，每步生成多个候选
2. **Thought Evaluation（思维评估）**：用 LLM 自身评估每个候选的质量（"这条路有希望吗？"）
3. **Search Algorithm（搜索算法）**：BFS（广度优先）或 DFS（深度优先）来遍历思维树

### 2.2 与传统搜索的对比

| 组件 | 传统搜索（如 A*） | Tree-of-Thought |
|------|-------------------|-----------------|
| 状态 | 棋盘、迷宫位置等 | 推理的中间文本 |
| 动作 | 移动、落子等 | 生成下一步推理 |
| 评估函数 | 手工设计的启发函数 | LLM 自我评估（"这一步好不好？"） |
| 搜索策略 | BFS / DFS / A* | BFS / DFS（由 LLM 驱动） |

### 2.3 经典案例：24 点游戏

**任务**：给定 4 个数字，用 +−×÷ 和括号组合得到 24。

```
输入: 1, 2, 3, 4

标准 CoT: 模型一次性生成一个表达式，成功率约 7.3%

ToT 流程:
第 1 步 — 生成候选操作（选两个数做运算）:
  候选 A: 1 + 2 = 3    (剩余: 3, 3, 4)
  候选 B: 3 × 4 = 12   (剩余: 1, 2, 12)
  候选 C: 2 × 3 = 6    (剩余: 1, 4, 6)
  候选 D: 1 × 4 = 4    (剩余: 2, 3, 4)

第 1 步 — 评估:
  候选 A: "3, 3, 4 → 可以 3 × (4+3)=21? 不太容易到 24" → 评分: 低
  候选 B: "1, 2, 12 → 12 × 2 = 24!" → 评分: 高 ✓
  候选 C: "1, 4, 6 → 4 × 6 = 24!" → 评分: 高 ✓
  候选 D: "2, 3, 4 → 2 × 3 × 4 = 24!" → 评分: 高 ✓

第 2 步 — 对高分候选继续展开...
```

**结果**：ToT 在 24 点任务上成功率 74%，远超 CoT 的 7.3%。

### 2.5 ToT 的局限

| 局限 | 说明 |
|------|------|
| API 调用次数多 | 每个节点需要"生成"和"评估"两次 LLM 调用，树越大越贵 |
| LLM 评估不准 | 模型评估"有没有希望"有时不靠谱，导致剪掉正确路径 |
| 任务适应性 | 效果在结构化问题（24 点、填字）上最好，开放式问题不太适用 |
| 延迟高 | 多轮搜索导致端到端延迟很高，不适合实时交互 |

## 三、Graph-of-Thought (GoT)

### 3.1 核心思想（Besta et al., 2023）

**论文**：*Graph of Thoughts: Solving Elaborate Problems with Large Language Models*（Besta et al., 2023）

ToT 是树形结构，有一个限制：**不同分支的信息不能合并**。但在实际推理中，两条思路的部分结论可以互相补充。

GoT 将推理结构从树扩展为**有向图**：

```
Tree (ToT):                    Graph (GoT):
    A                              A ←──┐
   / \                            / \    │
  B   C                          B   C   │ (合并/反馈)
 / \   \                        / \ / \  │
D   E   F                      D  E+F  G─┘

树：分支独立，不能合并         图：分支可合并，可回环
```

**GoT 新增的操作**：

| 操作 | 说明 | 示例 |
|------|------|------|
| **聚合（Aggregate）** | 合并多个思维节点的信息 | 把路径 A 和路径 B 的部分结论合并 |
| **精炼（Refine）** | 在已有节点上迭代改进 | 基于新信息修正之前的结论 |
| **回溯（Backtrack）** | 返回之前的节点重新开始 | 发现当前路径不对，退回 |

### 3.2 适用场景

GoT 在以下场景比 ToT 更有优势：
- **排序问题**：可以把数组分成多块分别排序，再合并
- **文本摘要**：可以对不同段落分别摘要，再聚合成最终摘要
- **代码审查**：分别分析不同模块的问题，再综合给出整体评估

## 四、Self-Reflection / Reflexion

### 4.1 核心思想（Shinn et al., 2023）

**论文**：*Reflexion: Language Agents with Verbal Reinforcement Learning*（Shinn et al., NeurIPS 2023）

CoT、ToT、GoT 都是在"同一次"推理中做优化。Reflexion 引入了一个新维度：**跨轮次的自我反思**。

```
Reflexion 循环:

轮次 1: 尝试解题 → 得到答案 → 检查：错了！
         ↓
         反思："我在第 3 步犯了计算错误，把乘法写成了加法"
         ↓
轮次 2: 重新解题（带着上轮的反思） → 得到答案 → 检查：对了！
```

**核心组件**：

1. **Actor（执行者）**：LLM，负责实际执行任务（写代码、解题等）
2. **Evaluator（评估者）**：判断结果是否正确（可以是测试用例、另一个 LLM、或人工反馈）
3. **Self-Reflection（自我反思）**：LLM 分析失败原因，生成文字形式的"经验教训"
4. **Memory（记忆）**：存储历史反思，在下一轮注入上下文

### 4.2 与人类学习的类比

| Reflexion 组件 | 人类类比 |
|----------------|----------|
| Actor | 考试时答题 |
| Evaluator | 老师批改 |
| Self-Reflection | 分析错题，写错题笔记 |
| Memory | 错题本，下次考试前翻看 |

### 4.3 Reflexion 在代码生成中的应用

Reflexion 在代码生成任务中效果尤其显著（HumanEval: 80.1% → 91.0%），因为代码有天然的评估器——单元测试。

概念流程可以概括为：

1. **生成**：模型根据任务描述给出一个候选解法。
2. **评估**：用测试、规则检查、另一个模型或人工反馈判断结果是否通过。
3. **反思**：如果失败，让模型用自然语言总结失败原因，例如“边界条件漏了”“推理第二步用了错误假设”。
4. **重试**：把反思内容加入下一轮上下文，让模型避免重复同样错误。

这套流程的关键不是“会写代码”，而是：**模型把失败经验显式写成文本，再把这段文本当作下一轮推理的额外记忆**。因此 Reflexion 也可以用于解题、网页任务、Agent 工具调用等场景，只要任务有某种评估信号。

### 4.4 Reflexion 的关键设计

```
                  ┌──────────────┐
                  │   任务描述    │
                  └──────┬───────┘
                         │
                         ▼
              ┌─────────────────────┐
              │  Actor (LLM 执行)   │ ← 注入历史反思
              └──────────┬──────────┘
                         │ 输出
                         ▼
              ┌─────────────────────┐
              │ Evaluator (测试/评估)│
              └──────────┬──────────┘
                         │
                    通过? ──── 是 ──→ 完成！
                         │
                         否
                         │
                         ▼
              ┌─────────────────────┐
              │ Self-Reflection     │
              │ (分析失败原因)       │
              └──────────┬──────────┘
                         │ 文字形式的"经验"
                         ▼
              ┌─────────────────────┐
              │  Memory (记忆存储)   │ → 注入下一轮 Actor
              └─────────────────────┘
```

## 五、ReAct：推理 + 行动

### 5.1 核心思想（Yao et al., 2023）

**论文**：*ReAct: Synergizing Reasoning and Acting in Language Models*（Yao et al., ICLR 2023）

ReAct 将推理（Reasoning）和行动（Acting）交替进行：

```
传统 CoT:    Thought → Thought → Thought → Answer
             (纯思考，不与外界交互)

传统 Acting: Action → Observation → Action → Observation
             (纯行动，不进行推理)

ReAct:       Thought → Action → Observation → Thought → Action → ...
             (想一步、做一步、看结果、再想再做)
```

### 5.2 经典示例

```
Question: 科罗拉多造山运动东部的什么山脉的最高峰海拔多少？

Thought 1: 我需要搜索科罗拉多造山运动，了解其东部有什么山脉。
Action 1: Search["科罗拉多造山运动"]
Observation 1: 科罗拉多造山运动影响了落基山脉东部的前山脉带...

Thought 2: 前山脉（Front Range）在科罗拉多造山运动的东部。我需要找前山脉的最高峰。
Action 2: Search["Front Range 最高峰"]
Observation 2: 前山脉的最高峰是 Grays Peak，海拔 4,352 米。

Thought 3: 我找到答案了。
Action 3: Finish["4,352 米"]
```

### 5.3 ReAct 与 Agent 的关系

ReAct 是 AI Agent 架构的理论基石。如果你后续学习模块 12（Agent 与工具使用），会看到 LangChain、LangGraph 等框架的核心循环就是 ReAct 模式的工程化实现。

```
ReAct (论文层面)                    Agent 框架 (工程层面)
━━━━━━━━━━━━━━━━                  ━━━━━━━━━━━━━━━━━━━
Thought                      →     LLM 推理 / Planning
Action                       →     Tool Use / Function Calling
Observation                  →     工具返回结果
Memory (Reflexion 扩展)      →     Agent 记忆模块
```

## 六、方法对比与选择指南

### 6.1 综合对比

| 方法 | 结构 | LLM 调用次数 | 最适合的任务 | 典型提升 |
|------|------|-------------|-------------|----------|
| CoT | 链 | 1 次 | 中等推理 | +15~20% (GSM8K) |
| Self-Consistency | 多条链 | N 次(5~40) | 有标准答案的推理 | +10~18% (GSM8K) |
| ToT | 树 | O(b^d) 次 | 结构化搜索问题 | +67% (24 点) |
| GoT | 图 | 灵活 | 可分解后合并的问题 | 因任务而异 |
| Reflexion | 多轮循环 | 多轮 × 每轮若干 | 有评估器的任务 | +11% (HumanEval) |
| ReAct | 交替链 | 多步 | 需要外部信息的问题 | 因任务而异 |

### 6.2 方法选择建议

在实际应用中，一条简洁的选择经验是：

> - **简单推理**（单步或少步推理）→ **CoT**（一句 "Let's think step by step" 即可）
> - **需要探索多条路径**（搜索式问题、组合优化）→ **ToT**（树形搜索 + 剪枝）
> - **需要自我纠错**（有明确评估器、允许多轮迭代）→ **Reflexion**（失败后反思 + 改进）
> - **需要工具调用**（依赖外部信息或 API）→ **ReAct**（推理 + 行动交替）

### 6.3 选择决策树

```
你的推理任务是什么类型？
│
├── 有标准答案的推理（数学/逻辑）
│   ├── 预算充足 → Self-Consistency (5~10 次采样)
│   ├── 需要搜索式探索 → ToT
│   └── 预算有限 → Zero-shot CoT
│
├── 需要与外部工具交互
│   └── ReAct（推理 + 行动循环）
│
├── 可以多轮迭代改进
│   └── Reflexion（尤其适合代码生成）
│
├── 可分解为子任务再合并
│   └── GoT（分别处理 + 聚合）
│
└── 简单直接的问题
    └── 不需要高级推理方法，直接回答
```

## 七、从提示方法到训练方法的转折

到目前为止，我们讨论的所有方法都有一个共同特点：**不改变模型参数**。

CoT、ToT、GoT、Reflexion、ReAct 都是在"如何提问"和"如何组织推理过程"上做文章。模型本身并不知道自己在做"树形搜索"或"自我反思"——这些都是外部框架在调度。

2024 年起，另一条路线取得了突破性进展：**直接训练模型学会推理**。

```
提示方法 (2022-2023):                  训练方法 (2024-2026):
外部框架控制推理过程                    模型自身学会推理过程
- CoT: 提示模型写步骤                  - o1: 模型自动产生长推理链
- ToT: 外部框架做搜索                  - R1: RL 训练模型"慢思考"
- Reflexion: 外部框架做反思循环        - GRPO: 群组策略优化训练推理

优点: 通用，不需训练                    优点: 更强，更自然，更可靠
缺点: 依赖外部编排，开销大              缺点: 需要专门训练，成本高
```

下一节将深入探讨这条"训练推理模型"的路线。

## 时效性说明

> - ToT、GoT 的原始论文发表于 2023 年，方法论已相对稳定
> - Reflexion 模式在 2024-2025 年被广泛集成到 Agent 框架中（LangGraph、CrewAI 等）
> - ReAct 已成为 Agent 架构的事实标准，在模块 12 中有更深入的工程化讨论
> - 截至 2026.4，随着推理模型（o1/o3/R1）的成熟，**纯提示方法的必要性正在降低**——推理模型本身已经内化了类似 ToT/Reflexion 的能力
> - 但在开源小模型和成本敏感场景中，提示方法仍有实用价值
