记忆与状态管理

前置知识：Agent 框架原理（ReAct 循环、Tool Use、Planning）
本节目标：理解 Agent 系统中记忆的三种类型——短期记忆、长期记忆、工作记忆——的原理与实现方式，掌握上下文窗口管理、向量存储、摘要压缩等核心技术。
内容时效：截至 2026 年 4 月

1. Agent 为什么需要记忆

1.1 核心问题：Agent 为什么需要"记忆"？

想象你正在处理一个复杂项目：

  • 你需要当前对话的上下文来理解同事刚说的话（短期记忆）
  • 你需要过去项目的经验教训来避免重复踩坑（长期记忆）
  • 你需要一张草稿纸来临时记录中间计算结果（工作记忆）

LLM 天然只有一种"记忆"——上下文窗口。但这个窗口有固定大小限制，且随着对话进行会不断增长直到溢出。一个实用的 Agent 系统必须解决：如何在有限的上下文窗口中，高效管理无限增长的信息？

1.2 记忆的三种类型

                        Agent 记忆系统
                             │
            ┌────────────────┼────────────────┐
            ▼                ▼                ▼
       短期记忆          长期记忆          工作记忆
    (Short-term)      (Long-term)      (Working Memory)
            │                │                │
    当前对话的上下文    跨会话持久化       当前任务的
    消息历史记录       的知识和经验       中间状态和计划
            │                │                │
    实现：上下文窗口    实现：向量数据库    实现：Scratch Pad
    生命周期：单次会话  /摘要/知识图谱     /状态变量/计划文档
                      生命周期：永久      生命周期：单次任务

1.3 类比人类认知

Agent 记忆类型人类认知类比特征
短期记忆工作记忆 / 对话记忆容量有限，当前正在处理的信息
长期记忆语义记忆 + 情景记忆容量大，需要检索才能使用
工作记忆草稿纸 / 白板临时存放中间结果，辅助当前推理

2. 短期、长期与工作记忆

2.1 短期记忆：上下文窗口管理

短期记忆就是 LLM 的上下文窗口——传入模型的完整 token 序列。这是 Agent 最基本也最关键的记忆形式。

上下文窗口的限制

模型上下文窗口约等于
GPT-4o（2024-2025）128K tokens~300 页文本
Claude 3.5/Opus 4（2025-2026）200K tokens~500 页文本
Gemini 1.5/2.0（2024-2026）1M-2M tokens~2500-5000 页文本

看似很大，但在 Agent 场景中消耗极快：

  • 系统 Prompt + 工具定义：2K-5K tokens
  • 每轮工具调用+返回：500-5000 tokens
  • 10 轮交互后：轻松超过 50K tokens
  • 长期运行的 Agent：几十轮后可能溢出

上下文管理策略

策略 1：滑动窗口（Sliding Window）

保留：System Prompt + 最近 N 轮对话
丢弃：最早的对话轮次

优点：实现简单
缺点：丢失早期重要信息

消息历史：[sys, u1, a1, u2, a2, u3, a3, u4, a4, u5, a5]
                              ↓ 窗口大小 = 3 轮
保留：      [sys,                         u3, a3, u4, a4, u5, a5]

策略 2：摘要压缩（Summarization）

保留：System Prompt + 历史摘要 + 最近 K 轮对话
做法：定期将旧对话压缩为摘要

消息历史：[sys, u1, a1, u2, a2, u3, a3, u4, a4, u5, a5]
                              ↓ 压缩前 3 轮为摘要
保留：      [sys, summary_of(u1~a3),      u4, a4, u5, a5]

策略 3：基于重要性的保留（Importance-based）

为每条消息计算重要性分数，优先保留高重要性消息：
- 用户的原始需求 → 高重要性
- 工具调用的关键结果 → 高重要性
- 中间的思考过程 → 低重要性
- 失败的尝试 → 中等重要性（含教训）

2.2 长期记忆：跨会话的持久化知识

长期记忆让 Agent 能够"记住"跨会话的信息——用户偏好、历史交互、学到的经验。

实现方案对比

方案原理优点缺点
向量存储将文本嵌入为向量，语义检索语义匹配强，模糊查询精确查询弱，需要维护索引
摘要压缩将历史压缩为结构化摘要紧凑，容易注入上下文压缩过程有信息损失
知识图谱提取实体和关系，构建图结构结构化，支持推理构建和维护成本高
键值存储按主题/类别存储为键值对检索快，精确匹配无语义泛化能力

向量存储的工作流程

写入流程：
    新信息 → Embedding 模型 → 向量 → 存入向量数据库（附带原始文本和元数据）

检索流程：
    查询 → Embedding 模型 → 查询向量 → 向量数据库相似度搜索 → Top-K 相关记忆
                                                                    ↓
                                                            注入到 Agent 上下文

记忆的生命周期管理

                 写入                            检索
新经验/信息 ─────────→ 长期记忆存储 ─────────→ 相关记忆
                          │
                     定期维护
                     ├── 去重：合并重复记忆
                     ├── 衰减：降低过时记忆的权重
                     ├── 整合：将碎片记忆合并为结构化知识
                     └── 遗忘：删除不再相关的记忆

2.3 工作记忆：Scratch Pad 与任务状态

工作记忆是 Agent 在执行当前任务时的"草稿纸"——存放中间结果、当前计划、待办事项等临时信息。

典型的工作记忆内容

工作记忆 vs 短期记忆

维度短期记忆（上下文）工作记忆（Scratch Pad）
内容完整对话历史提炼后的关键状态
格式自然语言对话结构化数据（JSON/表格）
更新追加式（新消息入队）覆盖式（状态更新）
作用维持对话连贯性辅助复杂任务推理

时效性说明

本节内容截至 2026 年 4 月。以下领域正在快速变化：

  • 上下文窗口持续扩大：Gemini 已支持 2M token，更大窗口可能减少对外部记忆的需求，但不会消除它——因为成本和延迟与窗口大小成正比
  • 模型原生记忆能力：各厂商正在将记忆管理内置到模型/产品层，框架层的记忆方案可能被逐步替代
  • 向量数据库生态：Pinecone、Weaviate、Qdrant、Chroma 等产品迭代迅速，API 和最佳实践以官方文档为准
