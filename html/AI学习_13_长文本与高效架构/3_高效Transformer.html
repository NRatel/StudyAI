高效 Transformer 变体

前置知识：注意力机制、Transformer 架构
内容时效：截至 2026 年 4 月

为什么有人想改造 Attention

标准 Self-Attention 的强大之处，是每个 token 都能直接看见其他 token。它的问题也正来自这里：序列越长，两两关系越多，计算和显存都按平方级增长。

100 个 token：大约 1 万个位置关系
1,000 个 token：大约 100 万个位置关系
10,000 个 token：大约 1 亿个位置关系

长文本场景下，很多研究都在问同一个问题：能不能少算一点，还尽量保留全局理解能力？

三条主要路线

路线直觉代价代表方向
稀疏注意力不让每个 token 看所有位置，只看附近和少数关键位置可能漏掉某些远距离关系Longformer、BigBird
线性注意力不显式生成巨大的 token × token 矩阵，改用可重排的近似计算近似会损失标准 softmax 注意力的一些性质Performer 等
状态空间模型不再两两比较 token，而是维护一个随序列更新的“状态”精确检索细节可能不如 Attention 直接S4、Mamba、RWKV

可以把三者理解成三种省钱方式：

  • 稀疏注意力：不是每个人都开会，只让相关的人开会。
  • 线性注意力：先做汇总，再按需查询，避免所有人两两对话。
  • 状态空间模型：边读边记笔记，用一份状态承载历史信息。

稀疏注意力：少看一点，但别断了联系

标准 Attention 像全员会议：每个 token 都能和所有 token 交流。稀疏注意力则像分组会议：大多数时候只看附近内容，少数关键位置负责连接全局。

标准注意力：
每个位置 -> 看所有位置

稀疏注意力：
每个位置 -> 看附近窗口
少数位置 -> 看全局
有些位置 -> 随机跨区连接

Longformer 的典型思路是“局部窗口 + 全局 token”：正文 token 主要看附近，像 [CLS] 或问题 token 这类特殊位置可以看全局。BigBird 在此基础上加入随机连接，让远距离信息更容易互相抵达。

这类方法适合长文档理解、分类、问答等任务，但它也有明显取舍：如果真正重要的信息刚好不在窗口、全局 token 或随机连接覆盖的位置，就可能被弱化。

线性注意力：别真的建那张巨大表

标准 Attention 会先形成一张“每个 token 对每个 token 有多相关”的大表。序列很长时，这张表本身就变成负担。

线性注意力的想法是：能不能换一种计算顺序，先把 Key/Value 做成某种汇总，再让 Query 去读取这个汇总？这样就不必显式构造那张巨大的两两关系表。

标准 Attention：先算所有位置两两关系，再加权汇总
线性 Attention：先压成可查询的汇总，再让每个位置去查

它的优点是理论上更省，尤其适合很长的序列。它的难点是：标准 softmax 注意力并不是随便能替换的，近似方法会影响质量，实际速度还会受到常数开销和硬件实现影响。

所以线性注意力很重要，但没有简单地“干掉”标准 Attention。今天更常见的结果是：它提供了思路，部分思想被吸收到混合架构或特定场景里。

SSM / Mamba：边读边更新状态

状态空间模型（SSM）走的是另一条路：不再让所有 token 两两比较，而是维护一个会随输入更新的隐藏状态。读到新 token 时，模型根据当前输入更新状态；生成或判断时，再从这个状态里取信息。

它有点像读书做笔记：

Attention：需要时翻回前文，直接看相关段落
SSM：一路读一路更新笔记，后面主要查笔记

这种思路天然适合长序列，因为它不必保存所有两两关系。Mamba 的关键改进，是让模型根据当前输入决定“哪些信息该记住，哪些信息可以淡化”。这比早期固定规则的状态更新更灵活，也让 SSM 在语言建模里重新受到关注。

不过，笔记再好也不是原文。对于需要精确回看某个细节的位置检索任务，Attention 的直接访问能力仍然很强。这也是为什么很多新架构不是完全替代 Transformer，而是采用混合方案：一部分层负责高效状态更新，一部分层保留注意力的精确读取能力。

为什么这些方案没有统一替代 Transformer

高效架构听起来很诱人，但真正落地会遇到几类问题：

问题含义
质量不一定稳少算或近似之后，某些任务会掉点
硬件不一定友好理论复杂度低，不代表 GPU 上一定更快
生态不一定成熟Transformer 的训练、推理、框架、工具链积累太深
任务需求不同长文档分类、代码补全、对话、检索增强，对架构的要求并不一样

因此更现实的判断是：标准 Transformer 仍是主干，高效变体提供补充路线。 在长文本、低延迟、边缘设备、超长序列建模等场景里，这些路线会继续影响新模型设计。

时效性说明

本文内容截至 2026 年 4 月。这一方向变化很快，阅读时可以重点关注三个问题：

  1. 新架构是否真的在主流任务上稳定优于 Transformer？
  2. 理论复杂度下降后，实际 GPU/推理系统里是否也更快？
  3. 它是替代 Transformer，还是成为混合架构里的一个部件？
