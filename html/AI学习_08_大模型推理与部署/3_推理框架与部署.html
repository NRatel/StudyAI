推理框架与部署

前置知识：第1章 KV Cache与推理优化、第2章 量化技术

框架选择速查：
- 生产环境高吞吐 → vLLM（PagedAttention + Continuous Batching，当前高并发推理的事实标准）
- NVIDIA GPU 极致性能 → TensorRT-LLM（编译优化 + 算子融合，延迟最低，但部署复杂度高）
- 本地 / CPU / 边缘设备 → llama.cpp（纯 C++ 实现 + GGUF 量化，7B Q4 模型只需 ~4GB 内存）

从模型能跑到服务能用

从"模型能跑"到"模型能用"

用 HuggingFace Transformers 加载一个模型，写两行代码就能生成文本。但这和"能在生产环境中服务用户"之间有巨大的鸿沟：

HuggingFace 推理（单请求）:
  - 吞吐量: ~30 tokens/s (7B, A100)
  - 并发: 1 个请求
  - 显存效率: 差（大量浪费）

vLLM 推理（高并发）:
  - 吞吐量: ~1500 tokens/s (7B, A100, batch=64)
  - 并发: 数十~数百个请求
  - 显存效率: 高（PagedAttention）

差距: 50x 的吞吐量

推理框架的核心工作就是把第 1、2 章学的优化技术（KV Cache、PagedAttention、Continuous Batching、量化）工程化，变成开箱即用的工具。

推理框架与部署技术

1. vLLM — 高吞吐推理的事实标准

核心技术栈

vLLM 之所以比朴素推理快 2~24 倍，是因为它集成了本模块前两章的几乎所有优化：

vLLM 技术栈:
  ├── PagedAttention (第1章)     → KV Cache 零浪费
  ├── Continuous Batching (第1章) → GPU 利用率最大化
  ├── 量化支持 (第2章)            → GPTQ, AWQ, FP8
  ├── CUDA Kernel 优化            → Flash Attention, 自定义算子
  ├── Tensor Parallelism          → 多 GPU 分布式推理
  └── Prefix Caching              → 系统 prompt 复用

vLLM 的调度器

vLLM 的调度器是其高吞吐的关键，它需要在每一步做出以下决策：

每一步 decode:
  1. 哪些请求已完成? → 移出 batch，释放 KV Cache 块
  2. 有新请求到达? → 尝试加入 batch
  3. 显存够吗? → 如果不够，是否需要 preempt（暂停某些请求）
  4. 有 prefill 请求? → 和 decode 请求混合调度

调度策略:
  - First-Come-First-Served (FCFS): 默认，公平
  - 优先短请求: 适合 SLA 敏感场景
  - Preemption: 显存不足时暂停低优先级请求，释放 KV Cache

2. TGI (Text Generation Inference) — HuggingFace 官方推理

TGI 是 HuggingFace 官方的推理框架，用 Rust 编写（高性能），核心特点：

特性说明
Flash Attention内置支持
Continuous Batching支持（称为 Token Streaming）
量化GPTQ、bitsandbytes、EETQ
分布式Tensor Parallelism
水印可对生成文本加水印
HF Hub 集成直接从 Hub 加载模型

TGI vs vLLM:

  • TGI 更适合已经深度使用 HuggingFace 生态的团队
  • vLLM 在纯吞吐量上通常领先（得益于 PagedAttention）
  • 两者都在快速迭代，差距在缩小

3. TensorRT-LLM — NVIDIA 的极致优化

TensorRT-LLM 是 NVIDIA 官方的 LLM 推理解决方案，核心是编译时优化：

HuggingFace/PyTorch 推理:
  Python → PyTorch → CUDA Kernels
  ↑ 大量 Python 开销，Kernel 未优化

TensorRT-LLM:
  模型定义 → TensorRT 编译 → 优化后的 CUDA Engine
  ↑ 消除 Python 开销，算子融合，显存优化

TensorRT-LLM 的核心优化：

优化说明
算子融合（Op Fusion）把多个小算子合并成一个大 Kernel，减少 Kernel 启动开销
FP8 量化在 H100/Ada 上利用硬件 FP8 单元
In-flight Batching类似 Continuous Batching，NVIDIA 版本
Paged KV Cache类似 PagedAttention
多 GPU 并行Tensor Parallel + Pipeline Parallel

优势：在 NVIDIA GPU 上通常能达到最低延迟。

劣势：编译过程复杂、对模型修改不灵活、与 NVIDIA 硬件深度绑定。

4. llama.cpp / GGML — CPU 推理的王者

llama.cpp（Georgi Gerganov, 2023）是一个纯 C/C++ 实现的 LLM 推理框架，专为低资源环境设计：

llama.cpp 核心特点:
  - 纯 C/C++，无 Python/PyTorch 依赖
  - 支持 CPU 推理（AVX2/ARM NEON 优化）
  - 也支持 GPU（CUDA/Metal/Vulkan）
  - GGUF 量化格式：2~8 bit 灵活选择
  - 极低内存占用：7B Q4 模型只需 ~4GB RAM

GGUF 格式的生态：

模型发布流程:
  原始模型 (FP16) → 转换为 GGUF → 提供多种量化版本

HuggingFace 上常见:
  TheBloke/Llama-2-7B-GGUF/
    ├── llama-2-7b.Q2_K.gguf    (2.87 GB, 最小)
    ├── llama-2-7b.Q4_K_M.gguf  (4.08 GB, 推荐)
    ├── llama-2-7b.Q5_K_M.gguf  (4.78 GB, 精度更好)
    ├── llama-2-7b.Q6_K.gguf    (5.53 GB, 接近无损)
    └── llama-2-7b.Q8_0.gguf    (7.16 GB, 几乎无损)

5. Prompt Engineering — 为什么提示词有效

Prompt Engineering 看起来像是"黑魔法"，但它的有效性有深层原因。

为什么提示词能影响模型输出？

根本原因：语言模型是条件概率生成器

$$P(\text{output} | \text{prompt}) = \prod_{t=1}^{T} P(x_t | x_1, ..., x_{t-1}, \text{prompt})$$

改变 prompt 就是改变了条件概率的条件，模型会到概率空间的不同区域去采样：

Prompt A: "写一首诗"
  → 模型进入"诗歌"概率区域
  → 输出押韵、有意境的文字

Prompt B: "作为一个资深 Python 开发者，解释这段代码"
  → 模型进入"技术专家"概率区域
  → 输出专业、结构化的解释

Prompt C: "一步一步地思考这个数学问题"
  → 模型进入"推理"概率区域 (Chain of Thought)
  → 输出详细的推理过程，而不是直接给答案

为什么 In-Context Learning 有效？

关于 ICL 为什么有效，有三个主流假说（详见模块 05）：

  1. 隐式贝叶斯推理：few-shot 示例帮助模型缩小任务空间
  2. 格式锚定：示例定义了输入-输出的格式，模型只需"模仿"
  3. 隐式梯度下降：注意力机制在前向传播中执行类似参数更新的操作

常见 Prompt 技巧及其原理

技巧原理效果
Chain of Thought (CoT)强制模型输出中间步骤，避免"跳步"数学推理准确率从 ~20% 提升到 ~80%
Few-shot在 prompt 中给出示例，锚定任务格式大幅提升任务准确率
Role Playing设定角色改变模型的先验分布输出风格和专业度改变
Self-Consistency多次采样取多数投票推理准确率进一步提升
ReAct交替 Reasoning 和 Acting让模型能使用外部工具

CoT 为什么对推理任务有效的深层原因

直接回答: "36 × 17 = ?"
  → 模型需要一步到位算出 612
  → 对于预训练时没见过的大数乘法，容易出错

CoT: "36 × 17, 让我们一步一步计算:"
  → "36 × 17 = 36 × 10 + 36 × 7"
  → "36 × 10 = 360"
  → "36 × 7 = 252"
  → "360 + 252 = 612"
  → 每一步都是简单运算，大模型都能做对
  → 相当于把难题分解成多个简单题

本质: CoT 把"一步推理"变成"多步序列化推理"
      而自回归模型天然擅长序列化生成
