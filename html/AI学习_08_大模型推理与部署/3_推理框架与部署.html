推理框架与部署

前置知识：第1章 KV Cache与推理优化、第2章 量化技术

从“模型能跑”到“服务能用”

把模型加载起来生成一句话，只说明它能跑；让它稳定服务很多用户，是另一回事。推理服务真正关心四个问题：

问题具体含义常见优化
首字慢不慢用户多久看到第一个 tokenprefill 优化、缓存、调度
连续输出快不快每秒能吐出多少 tokenKV Cache、批处理、算子优化
能同时服务多少人多请求并发时是否堵住Continuous Batching、PagedAttention
成本能不能接受显存、GPU 数量、电费是否可控量化、推测解码、本地/边缘推理

推理框架的价值，就是把 KV Cache、分页管理、批处理、量化、算子优化这些技术组合起来，让模型从“单次调用能生成”变成“多人同时用也扛得住”。

推理框架不是主角，瓶颈才是主角

不同框架看起来名字很多，其实是在偏向不同瓶颈：

路线主要解决什么代表工具先记住什么
高吞吐在线服务很多请求同时来，GPU 不能闲着vLLM把 KV Cache 管好，把请求动态塞进 batch
生态集成已经在 HuggingFace 体系里训练和发布模型TGI少折腾集成，直接接入 HF 模型生态
硬件极限优化在特定 NVIDIA GPU 上榨低延迟TensorRT-LLM编译、融合、硬件适配带来速度，但灵活性下降
本地和边缘运行没有大 GPU，也想跑模型llama.cpp / Ollama量化和轻量运行时让小模型进入普通设备

第一次读到这里，不需要背框架功能列表。更重要的是看懂：同一个模型，在不同场景里瓶颈不同，所以工具路线也不同。

vLLM：把“等车的人”组织成一车一车发

自回归生成每一步只产出一个 token。单个请求会让 GPU 很多时候吃不饱；多个请求同时来，又会因为长度不同、KV Cache 大小不同而难以管理。

vLLM 的核心直觉可以拆成两件事：

  1. Continuous Batching：不要等一批请求全部结束才处理下一批，而是在生成过程中不断把新请求塞进来、把完成的请求拿出去。
  2. PagedAttention：把 KV Cache 切成块管理，像操作系统分页一样，需要多少用多少，减少显存碎片。
朴素方式：一批请求排队，短请求等长请求，显存还容易碎
vLLM 思路：请求动态上下车，KV Cache 分块管理，GPU 尽量保持忙碌

所以 vLLM 的重点不是“它有多少 API”，而是它把推理服务里最麻烦的两个问题放在了一起处理：调度请求和管理 KV Cache。

TensorRT-LLM、TGI、llama.cpp 分别在补哪块

TensorRT-LLM：把运行过程提前编译好

普通 PyTorch 推理更灵活，但运行时开销较多。TensorRT-LLM 的思路是提前针对模型和硬件做编译优化，把多个小操作融合成更适合 GPU 执行的大操作。

它适合追求低延迟和高性能的场景，但代价是部署链路更复杂，对硬件和模型形态更挑剔。

TGI：让 HuggingFace 生态里的模型更容易服务化

TGI 的重点不是提出一个全新的原理，而是把模型加载、流式输出、批处理、量化支持、监控等服务化能力打包好。对于已经深度使用 HuggingFace 的团队，它降低的是接入成本。

llama.cpp / Ollama：让模型离开数据中心

本地推理关心的问题不同：没有大 GPU，显存和内存都有限，速度也不能太差。llama.cpp 这类工具依靠轻量运行时和 GGUF 量化格式，让较小模型能在 CPU、笔记本、边缘设备上运行。

这类路线的意义不是替代云端大模型，而是让私有、离线、低成本的小模型场景变得可行。

Prompt Engineering 为什么放在推理部署里

Prompt Engineering 看起来像写话术，但从模型角度看，它是在改变生成条件。语言模型每一步都在根据已有上下文预测下一个 token；你改了 prompt，就改变了模型“当前处在什么任务、什么语气、什么格式里”。

“写一首诗”
  → 模型更容易进入诗歌风格

“用三句话解释给产品经理听”
  → 模型更容易控制长度和受众

“一步一步分析”
  → 模型更可能把中间推理过程展开

Prompt 的有效性不神秘：它不是改了模型参数，而是改了模型看到的上下文。它的局限也在这里：如果模型没有相关能力，prompt 只能引导，不能凭空创造能力。

CoT 的直觉

Chain-of-Thought 的核心不是“咒语”，而是把一步到位的难题拆成一串更小的中间步骤。自回归模型天然擅长生成序列，把复杂推理序列化后，模型更容易沿着步骤往下走。

但 CoT 也不是万能的：它可能提高推理题表现，也可能让模型更自信地展开错误过程。是否使用 CoT，要看任务是否真的需要多步推理。

本章小结

概念一句话总结
推理服务关心延迟、吞吐、显存、成本，不只是模型能不能生成
vLLM用动态批处理和分页式 KV Cache 管理提高高并发吞吐
TGI把 HuggingFace 模型更顺地变成可服务化接口
TensorRT-LLM用编译和硬件适配换更低延迟，代价是复杂度和灵活性
llama.cpp / Ollama依靠量化和轻量运行时，把模型带到本地和边缘设备
Prompt Engineering改变上下文条件，引导模型进入合适的任务和输出模式
CoT把复杂推理拆成序列化步骤，但不保证每一步都正确
